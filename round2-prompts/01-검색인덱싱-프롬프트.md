# 작업 지시: 내 도메인 데이터로 OpenSearch 하이브리드 검색 구축

너는 지금 **검색 인덱싱 파이프라인이 아직 없는** 프로젝트 안에 있다.
이 프로젝트에는 이미 5개 도메인의 **수집 완료된 데이터**가 `poc/data/<도메인>/`
에 들어있다. 아래 명세대로 그중 **수강생이 택한 1개 도메인**의 검색 문서를
OpenSearch에 인덱싱하고, **검색 3종(BM25 / 벡터 / 하이브리드)** 을 실행해
top-5 결과를 비교 출력하는 파이프라인을 **끝까지** 만들어라.

이 지시서는 코드를 짤 줄 모르는 사람이 너에게 그대로 붙여넣은 것이다.
명세가 완결돼 있으니 **너 스스로 끝까지 수행**하고, 진행 불가 상황(아래
§0-5)만 사용자에게 보고한다. 그 외에는 질문하지 마라.

---

## 0. 절대 규칙 (위반 시 작업 실패)

> 아래 6개는 우리가 사전 검증으로 알아낸 **함정**이다. 하나라도 어기면
> 인덱스를 통째로 재생성해야 하거나, 강의 데모가 무너진다. 예외 없다.

1. **데이터는 이미 있다 — 재수집·재크롤링 절대 금지.**
   `poc/data/<도메인>/`에 **구조화 CSV**와 **검색 문서 JSONL**이 수집
   완료돼 있다. 너는 이걸 **읽어서 인덱싱만** 한다. 새로 크롤링하거나
   API를 호출해 문서를 더 모으는 행위, 기존 데이터 파일을 덮어쓰거나
   수정하는 행위 **일절 금지**. 도메인은 아래 5개 중 **수강생이 지정한
   정확히 1개**다:

   | 도메인 키 | 내용 | 검색 문서 JSONL |
   |---|---|---|
   | `sangkwon` | 상권/소상공인 (정책뉴스 소상공인 서브셋) | `poc/data/sangkwon/policy_news.jsonl` |
   | `medical`  | 의료/제약 (의약품 허가상세) | `poc/data/medical/drug_detail.jsonl` |
   | `finance`  | 금융 (정책뉴스 금융 서브셋) | `poc/data/finance/policy_news.jsonl` |
   | `legal`    | 법률 (법령 본문 코퍼스) | `poc/data/legal/law_corpus.jsonl` |
   | `policy`   | 정책 (부처 보도/정책자료) | `poc/data/policy/press_release.jsonl` |

   - 수강생이 도메인을 안 줬으면 **그때만** "5개 중 어느 도메인인가요?"
     1회 물어보고 답을 기다린다. 줬으면 묻지 말고 그 도메인만 처리한다.
   - 다른 4개 도메인 데이터는 **건드리지 마라**.

2. **knn 차원은 비가역 락인이다 — 중간에 임베딩 모델 절대 변경 금지.**
   OpenSearch `knn_vector` 필드의 `dimension`은 **인덱스를 만드는 순간
   고정**된다. 나중에 임베딩 모델을 바꾸면 차원이 안 맞아 **인덱스
   전체를 삭제하고 재생성·재인덱싱**해야 한다(강의 D-day엔 치명적).
   - 임베딩은 **OpenAI `text-embedding-3-small`(1536차원)** 으로 시작
     시점에 **고정 결정**한다. 한 번 정하면 이 작업이 끝날 때까지
     절대 다른 모델·차원으로 바꾸지 않는다.
   - **OpenAI API 키가 없으면**(`OPENAI_API_KEY` 환경변수 부재):
     인덱스를 만들기 **전에 멈추고** 사용자에게 "OpenAI API 키
     (`OPENAI_API_KEY`)가 없어 임베딩·벡터·하이브리드 검색을
     진행할 수 없습니다. 키를 설정해 주세요"라고 **보고하고 종료**한다.
     **임의의 다른 임베딩 모델로 fallback 금지**(허락 없이 로컬
     모델·다른 차원으로 바꾸면 §0-2 위반).

3. **청킹 강제 — 통짜 임베딩 절대 금지.**
   문서를 임베딩하기 전 반드시 **1000 토큰 슬라이딩 윈도우, overlap
   100 토큰**으로 청킹한다. 단일 문서가 아무리 길어도(예: 의약품
   허가상세 본문은 27,000자에 달함) **문서 전체를 한 번에 임베딩하지
   마라**. 길이와 무관하게 모든 문서는 청킹을 거친다. 토큰 계산은
   임베딩 모델 토크나이저(예: `tiktoken`의 `cl100k_base`)로 한다.

4. **컨테이너 라이프사이클은 run 스크립트가 책임진다.**
   OpenSearch는 **Docker 컨테이너 1개**로 로컬(`localhost:9200`)에
   띄운다. 프로젝트 루트에 `run-search.sh`를 만들어 거기서:
   기존 동명 컨테이너 **kill/rm → 기동 → 헬스체크 폴링(`sleep`
   금지, `curl` + `until` 루프) → 파이프라인 실행 → 종료 시 정리**
   (`trap`으로 컨테이너 stop)를 **전부** 관리한다. 파이썬 코드 안에서
   `docker run`을 호출하지 마라(라이프사이클·연결 책임 분리).
   - OpenSearch **데이터 볼륨은 `poc/` 하위 경로**에만 둔다(예:
     `poc/opensearch/data/`). 루트나 임의 경로에 두지 마라.

5. **사용자에게 불필요한 질문 금지.** 명세가 완결돼 있다. 진행을
   막는 상황 **2가지만** 보고한다: ① 도메인 미지정(§0-1), ②
   OpenAI 키 부재(§0-2). 그 외 모호함은 이 문서 문구를 그대로 따른다.

6. **산출물은 커밋 금지.** 이 프로젝트는 공개 배포 리포다. 인덱스
   데이터·임베딩·OpenSearch 볼륨·코퍼스 JSONL은 제3자 저작권 본문
   역추출이 가능하므로 **절대 git에 들어가면 안 된다**. 작업 전
   `.gitignore`에 아래가 차단돼 있는지 확인하고, 없으면 추가한다
   (커밋하라는 말이 아니라 *제외*돼 있는지 확인하라는 것):
   `poc/data/`, `*.jsonl`, `poc/opensearch/data/`, `poc/**/index/`,
   `.env`, `*.faiss`, `*.index`. **버전관리 OK인 것은 코드와
   `poc/opensearch/mapping.json`(매핑 스키마)뿐**이다.

## 1. 무엇을 만드나

수강생이 택한 **1개 도메인**의 검색 문서로 OpenSearch **하이브리드
검색 PoC**를 구축한다. 최종적으로 같은 질의에 대해 **3가지 검색 방식**의
top-5 결과를 나란히 보여줘서, 세 방식이 어떻게 다르게 동작하는지
사람이 눈으로 비교할 수 있게 한다.

만들 산출물:

| 경로 | 역할 | git 커밋 |
|---|---|---|
| `run-search.sh` (루트) | 컨테이너 라이프사이클 + 파이프라인 오케스트레이터 | O (코드) |
| `poc/opensearch/mapping.json` | 인덱스 매핑·analyzer·knn 설정 (버전관리 스키마) | O (스키마) |
| `poc/search/index_build.py` | JSONL 로드 → 청킹 → 임베딩 → bulk 인덱싱 | O (코드) |
| `poc/search/search_compare.py` | 검색 3종 실행 + top-5 비교 출력 | O (코드) |
| `poc/search/queries_<도메인>.txt` | 도메인별 질의 세트(설계형+자연어형) | O (코드) |
| `poc/opensearch/data/` | OpenSearch 데이터 볼륨 | **X (제외)** |
| 인덱싱된 인덱스·임베딩 | 런타임 산출물 | **X (제외)** |

검색 3종 (반드시 3개 모두 실행):

- **(a) BM25 (Nori)** — 키워드/형태소 기반. `multi_match`로 Nori·ngram
  분석 필드를 검색. 정확한 용어·법조문 번호·고유명사에 강함.
- **(b) 벡터 (knn)** — 의미 기반. 질의를 OpenAI 임베딩으로 변환해
  `knn` 검색. 표현이 달라도 의미가 가까우면 잡음.
- **(c) 하이브리드 (OpenSearch 네이티브)** — `multi_match`(BM25) +
  `knn`(boost)를 **단일 `bool.should` 쿼리** 안에 결합. boost
  가중치를 grid로 바꿔가며 최적 조합을 찾고 채택값을 출력.
  **RRF 등 별도 융합 로직을 직접 구현하지 마라** — OpenSearch
  네이티브 단일 쿼리 결합만 사용.

## 2. 수행 절차 (이 순서대로 — 건너뛰기 금지)

1. **사전 점검 (Step 0)** — 도메인 확정, OpenAI 키 확인, `.gitignore`
   확인, Docker 가용 확인. 진행 불가면 여기서 보고·종료.
2. **`poc/opensearch/mapping.json` 생성 (Step 1)** — §3 매핑을
   그대로 쓴다. knn 차원은 §0-2에서 고정한 **1536**.
3. **`run-search.sh` 생성 (Step 2)** — §4 라이프사이클 스크립트.
4. **`poc/search/index_build.py` 생성 (Step 3)** — §5 인덱싱
   파이프라인 (로드→청킹→임베딩→bulk).
5. **`poc/search/queries_<도메인>.txt` 생성 (Step 4)** — §6 질의 세트.
6. **`poc/search/search_compare.py` 생성 (Step 5)** — §7 검색 3종 비교.
7. **`run-search.sh` 실행 (Step 6)** — 컨테이너 기동→인덱싱→검색3종.
8. **§검증/완료 기준** 을 표로 보고하고 종료.

### Step 0 — 사전 점검 (가장 먼저, 빠짐없이)

```bash
# (0-1) 도메인 확정 — 수강생이 준 값. 안 줬으면 1회 질문.
DOMAIN="<sangkwon|medical|finance|legal|policy 중 1개>"

# (0-2) 검색 문서 JSONL 경로 (도메인별 고정 — §0-1 표)
case "$DOMAIN" in
  sangkwon) DOC_JSONL="poc/data/sangkwon/policy_news.jsonl" ;;
  medical)  DOC_JSONL="poc/data/medical/drug_detail.jsonl" ;;
  finance)  DOC_JSONL="poc/data/finance/policy_news.jsonl" ;;
  legal)    DOC_JSONL="poc/data/legal/law_corpus.jsonl" ;;
  policy)   DOC_JSONL="poc/data/policy/press_release.jsonl" ;;
  *) echo "FAIL: 알 수 없는 도메인 '$DOMAIN'"; exit 1 ;;
esac

# (0-3) 데이터 존재 확인 — 없으면 재수집 금지, 사용자에게 보고
[ -f "$DOC_JSONL" ] || { echo "FAIL: $DOC_JSONL 없음. 데이터가 수집돼 있어야 함(재수집 금지)"; exit 1; }
echo "문서 수: $(wc -l < "$DOC_JSONL") 건"

# (0-4) OpenAI 키 확인 — 없으면 §0-2대로 멈추고 보고
[ -n "$OPENAI_API_KEY" ] || { echo "STOP: OPENAI_API_KEY 없음. 임베딩 진행 불가 — 사용자에게 보고 후 종료(임의 fallback 금지)"; exit 2; }

# (0-5) Docker 가용
docker info >/dev/null 2>&1 || { echo "FAIL: Docker 데몬이 떠 있지 않음"; exit 1; }

# (0-6) .gitignore 차단 확인 (없으면 추가만, 커밋 안 함)
for pat in 'poc/data/' '*.jsonl' 'poc/opensearch/data/' '.env'; do
  grep -qF "$pat" .gitignore 2>/dev/null || echo "$pat" >> .gitignore
done
```

검색 문서 JSONL **1줄 = 1문서**, 각 줄은 JSON 객체이며 **최소 필드**:

```json
{"doc_id": "policy_148964305", "title": "문서 제목", "body": "본문 전체 텍스트...", "url": "https://...", "char_count": 2027}
```

(도메인에 따라 추가 필드가 더 있을 수 있다 — `medical`은 `item_seq`,
`legal`은 `law_mst`/`ministry`/`promulgation_date` 등. 검색·인덱싱에
쓰는 핵심은 `doc_id`/`title`/`body`/`url`/`char_count` 5개이며, 그
외 필드는 있으면 keyword 메타로 같이 넣고 없으면 무시한다. **스키마가
다르다고 데이터를 다시 만들지 마라**.)

### Step 1 — 인덱스 매핑 생성 `poc/opensearch/mapping.json`

> 이 매핑은 사내 운영 인덱스(`requirement/커뮤니티-AI검색/04-opensearch-index.md`
> §7)의 검증된 구조를 이 PoC 스키마(`doc_id/title/body/url`)와 OpenAI
> 임베딩 차원(**1536**, `text-embedding-3-small`)에 맞춰 적용한 것이다.
> **knn `dimension`은 1536으로 고정** — §0-2에 따라 절대 바꾸지 마라.

`poc/opensearch/mapping.json` 에 아래 JSON을 **그대로** 쓴다:

```json
{
  "settings": {
    "index": {
      "number_of_shards": 1,
      "number_of_replicas": 0,
      "knn": true,
      "knn.algo_param.ef_search": 100,
      "max_ngram_diff": 6
    },
    "analysis": {
      "tokenizer": {
        "nori_tokenizer": {
          "type": "nori_tokenizer",
          "decompound_mode": "mixed"
        },
        "ngram_tokenizer": {
          "type": "ngram",
          "min_gram": 1,
          "max_gram": 2,
          "token_chars": ["letter", "digit"]
        }
      },
      "analyzer": {
        "nori_analyzer": {
          "type": "custom",
          "tokenizer": "nori_tokenizer",
          "filter": ["nori_readingform", "nori_part_of_speech", "lowercase", "trim"]
        },
        "ngram_analyzer": {
          "type": "custom",
          "tokenizer": "ngram_tokenizer",
          "filter": ["lowercase", "trim"]
        }
      }
    }
  },
  "mappings": {
    "dynamic": "true",
    "properties": {
      "doc_id":     {"type": "keyword"},
      "chunk_id":   {"type": "keyword"},
      "title": {
        "type": "text",
        "analyzer": "nori_analyzer",
        "fields": {
          "ngram": {"type": "text", "analyzer": "ngram_analyzer"}
        }
      },
      "body": {
        "type": "text",
        "analyzer": "nori_analyzer",
        "fields": {
          "ngram": {"type": "text", "analyzer": "ngram_analyzer"}
        }
      },
      "url":        {"type": "keyword"},
      "char_count": {"type": "long"},
      "chunk_text": {
        "type": "text",
        "analyzer": "nori_analyzer",
        "fields": {
          "ngram": {"type": "text", "analyzer": "ngram_analyzer"}
        }
      },
      "vector_field": {
        "type": "knn_vector",
        "dimension": 1536,
        "method": {
          "name": "hnsw",
          "space_type": "cosinesimil",
          "engine": "faiss",
          "parameters": {"ef_construction": 256, "m": 16}
        }
      }
    }
  }
}
```

설명 (에이전트 이해용 — 파일엔 위 JSON만 쓴다):

- **Nori analyzer** (`decompound_mode=mixed`): 한국어 복합어를
  분해해 색인. "중대재해처벌법" → "중대재해" + "처벌법" + 원형 모두
  토큰화 → 정확 키워드 검색 강화. `nori_part_of_speech`로 조사·어미
  같은 불용 품사 제거.
- **ngram analyzer** (1~2gram): 형태소 분해가 놓치는 부분일치를
  커버(오타·신조어·부분어 검색). `title`/`body`/`chunk_text` 모두
  `.ngram` 서브필드를 둬 둘을 동시에 검색한다.
- **knn_vector** (`dimension: 1536`, HNSW, `faiss`, `cosinesimil`):
  OpenAI `text-embedding-3-small` 임베딩 저장 필드. **차원은 임베딩
  모델이 결정** — 1536 고정. `m=16`, `ef_construction=256`은 운영
  검증값.
- `number_of_replicas: 0`: 단일 노드 로컬 컨테이너라 replica 불필요
  (운영은 1, PoC는 0이 정상 — yellow health 회피).

### Step 2 — 라이프사이클 스크립트 `run-search.sh` (프로젝트 루트)

> §0-4: 컨테이너 기동·헬스체크·정리는 **전부 이 스크립트**가 책임진다.
> `sleep` 금지 — `curl` + `until` 폴링. 종료 시 `trap`으로 컨테이너 정지.

`run-search.sh` 에 아래를 쓴다 (실행권한: `chmod +x run-search.sh`):

```bash
#!/usr/bin/env bash
set -euo pipefail

# ── 설정 ──────────────────────────────────────────────
DOMAIN="${1:?사용법: ./run-search.sh <sangkwon|medical|finance|legal|policy>}"
CONTAINER="aiceo-opensearch"
OS_PORT=9200
OS_IMAGE="opensearchproject/opensearch:2.13.0"
ROOT="$(cd "$(dirname "$0")" && pwd)"
```

### Step 3 — 인덱싱 파이프라인 `poc/search/index_build.py`

JSONL 로드 → **청킹(1000토큰/overlap100)** → **OpenAI 임베딩** →
**bulk 인덱싱**. 아래 골격대로 작성한다 (네 환경의 라이브러리 버전에
맞게 import·호출만 조정, **로직·상수는 유지**):

```python
import argparse, json, os, sys, hashlib
from pathlib import Path
import tiktoken
from openai import OpenAI
from opensearchpy import OpenSearch, helpers

DOC_PATHS = {
    "sangkwon": "poc/data/sangkwon/policy_news.jsonl",
    "medical":  "poc/data/medical/drug_detail.jsonl",
    "finance":  "poc/data/finance/policy_news.jsonl",
    "legal":    "poc/data/legal/law_corpus.jsonl",
    "policy":   "poc/data/policy/press_release.jsonl",
}
EMBED_MODEL = "text-embedding-3-small"   # §0-2 비가역 고정. 절대 변경 금지.
EMBED_DIM   = 1536                       # 매핑 dimension과 단일 출처로 일치.
CHUNK_TOKENS, OVERLAP = 1000, 100        # §0-3 강제. 통짜 임베딩 금지.

def chunk(text, enc):
    """1000토큰 슬라이딩 윈도우, overlap 100. 길이 무관 전 문서 적용."""
    toks = enc.encode(text or "")
    if not toks:
        return []
    out, start = [], 0
    while start < len(toks):
        window = toks[start:start + CHUNK_TOKENS]
        out.append(enc.decode(window))
        if start + CHUNK_TOKENS >= len(toks):
            break
        start += CHUNK_TOKENS - OVERLAP
    return out

def main():
    ap = argparse.ArgumentParser()
    ap.add_argument("--domain", required=True, choices=list(DOC_PATHS))
    ap.add_argument("--host", required=True)
    args = ap.parse_args()

    api_key = os.environ.get("OPENAI_API_KEY")
    if not api_key:                       # §0-2: 키 없으면 멈추고 보고. fallback 금지.
        print("STOP: OPENAI_API_KEY 없음 — 임베딩 불가. 사용자 보고 후 종료.")
        sys.exit(2)

    doc_path = Path(DOC_PATHS[args.domain])
    if not doc_path.exists():             # §0-1: 재수집 금지. 없으면 실패 보고.
        print(f"FAIL: {doc_path} 없음 (데이터 수집 완료 전제 — 재수집 금지)")
        sys.exit(1)

    enc = tiktoken.get_encoding("cl100k_base")
    oai = OpenAI(api_key=api_key)
    os_client = OpenSearch(hosts=[args.host])
    index = f"aiceo-{args.domain}"

    # 인덱스 (재)생성 — 매핑은 버전관리된 mapping.json 사용
    mapping = json.loads(Path("poc/opensearch/mapping.json").read_text())
    assert mapping["mappings"]["properties"]["vector_field"]["dimension"] == EMBED_DIM, \
        "매핑 dimension과 EMBED_DIM 불일치 — knn 차원 단일 출처 위반"
    if os_client.indices.exists(index):
        os_client.indices.delete(index)
    os_client.indices.create(index=index, body=mapping)

    # 로드 → 청킹 → 임베딩 → bulk
    actions = []
    n_docs = n_chunks = 0
    for line in doc_path.read_text(encoding="utf-8").splitlines():
        line = line.strip()
        if not line:
            continue
        d = json.loads(line)
        n_docs += 1
        full = f"{d.get('title','')}\n\n{d.get('body','')}"
        chunks = chunk(full, enc)         # 통짜 금지 — 항상 청킹 경유
        if not chunks:
            continue
        # OpenAI 임베딩은 배치 호출 (한 번에 여러 청크). 모델·차원 고정.
        for i in range(0, len(chunks), 64):
            batch = chunks[i:i+64]
            resp = oai.embeddings.create(model=EMBED_MODEL, input=batch)
            for j, emb in enumerate(resp.data):
                vec = emb.embedding
                assert len(vec) == EMBED_DIM, f"임베딩 차원 {len(vec)} != {EMBED_DIM}"
                cid = f"{d['doc_id']}#${i+j}"
                actions.append({
                    "_index": index,
                    "_id": hashlib.md5(cid.encode()).hexdigest(),
                    "_source": {
                        "doc_id": d["doc_id"],
                        "chunk_id": cid,
                        "title": d.get("title", ""),
                        "body": d.get("body", ""),
                        "url": d.get("url", ""),
                        "char_count": d.get("char_count", len(d.get("body", ""))),
                        "chunk_text": batch[j],
                        "vector_field": vec,
                    },
                })
                n_chunks += 1
        # bulk를 5000 액션마다 flush (메모리 보호)
        if len(actions) >= 5000:
            helpers.bulk(os_client, actions); actions = []
    if actions:
        helpers.bulk(os_client, actions)
    os_client.indices.refresh(index=index)

    cnt = os_client.count(index=index)["count"]
    print(f"인덱싱 완료: 도메인={args.domain} 문서={n_docs} 청크={n_chunks} "
          f"인덱스문서={cnt} 모델={EMBED_MODEL}({EMBED_DIM}d)")

if __name__ == "__main__":
    main()
```

> 임베딩 호출은 **반드시 배치**(`input=[...]`). 문서 1개라도 청킹 후
> 청크 단위로 임베딩 — **문서 통짜 임베딩 절대 금지**(§0-3).
> `EMBED_MODEL`/`EMBED_DIM`은 한 곳에서만 정의해 매핑과 단일 출처로
> 일치시킨다(`assert`로 강제). 도중에 바꾸면 §0-2 위반.

### Step 4 — 질의 세트 `poc/search/queries_<도메인>.txt`

각 줄 1질의. **절반은 설계형**(BM25/벡터/하이브리드 우세가 의도된
질의), **절반은 수강생이 던질 법한 거친 자연어형**. 6~10개.
**수강생이 택한 도메인의 파일만** 만든다(`<도메인>`을 실제 키로 치환).

도메인별 예시 (그대로 써도 되고 도메인 성격에 맞게 다듬어도 됨):

- `sangkwon` / `finance` / `policy` (정책뉴스·보도자료 계열):
  ```
  중대재해처벌법 시행령
  소상공인 손실보상 신청 기한
  2026년 소상공인 R&D 세액공제 변경
  기업이 친환경 전환할 때 받는 정부 지원
  자영업자 대출 갈아타기 정책
  요즘 물가 잡으려고 정부가 뭐 한대?
  전통시장 살리기 예산 얼마야
  창업하면 세금 깎아주는 거 있어?
  ```
- `medical` (의약품 허가상세):
  ```
  아세트아미노펜 효능효과
  임부 금기 주사제
  소아 용법용량 조절이 필요한 항생제
  간 기능 저하 환자에게 주의해야 하는 약
  이 약 임신했을 때 먹어도 돼?
  콩팥 안 좋으면 조심해야 하는 주사 뭐 있어
  ```
- `legal` (법령 본문):
  ```
  건축물의 설비기준 등에 관한 규칙 제2조
  관계전문기술자 협력 의무 건축물
  개인정보 파기 의무 위반 시 제재
  세입자가 보증금 못 돌려받을 때 보호받는 법
  회사가 직원 해고할 때 지켜야 하는 절차
  소방시설 설치 안 하면 어떻게 돼?
  ```

설계형은 "정확한 용어/조문번호 → BM25 우세", "의미는 같지만 표현이
다른 자연어 → 벡터 우세", "정확 용어 + 맥락이 섞임 → 하이브리드
우세"가 갈리도록 의도해 구성한다.

### Step 5 — 검색 3종 비교 `poc/search/search_compare.py`

질의마다 **BM25 / 벡터 / 하이브리드** 를 각각 실행하고 **top-5**를
나란히 출력한다. 하이브리드는 **boost grid**를 돌려 채택값을 명시한다.

> 아래 3개 쿼리 본문은 사내 운영 패턴
> (`requirement/커뮤니티-AI검색/04-opensearch-index.md` §3 하이브리드
> 쿼리·§4 필드 가중치)을 이 PoC 스키마에 맞춘 것이다. **추측하지 말고
> 이 쿼리 구조를 그대로** 쓴다.

```python
import argparse, os, sys
from openai import OpenAI
from opensearchpy import OpenSearch

EMBED_MODEL, EMBED_DIM = "text-embedding-3-small", 1536  # index_build와 단일 출처 일치

# (a) BM25 — Nori + ngram 필드. 정확 용어·조문번호에 강함.
def q_bm25(qtext):
    return {
        "size": 5,
        "_source": ["doc_id", "title", "url"],
        "query": {
            "multi_match": {
                "query": qtext,
                "fields": ["title^4", "title.ngram^2",
                           "body^2", "body.ngram",
                           "chunk_text^2", "chunk_text.ngram"],
                "type": "best_fields",
                "tie_breaker": 0.3
            }
        }
    }

# (b) 벡터 — knn 의미 검색.
def q_vector(qvec):
    return {
        "size": 5,
        "_source": ["doc_id", "title", "url"],
        "query": {"knn": {"vector_field": {"vector": qvec, "k": 40}}}
    }

# (c) 하이브리드 — multi_match(BM25) + knn(boost) 단일 bool.should.
#     RRF 등 별도 융합 구현 금지 — OpenSearch 네이티브 단일 쿼리 결합.
def q_hybrid(qtext, qvec, knn_boost, title_boost):
    return {
        "size": 5,
        "_source": ["doc_id", "title", "url"],
        "query": {
            "bool": {
                "should": [
                    {"multi_match": {
                        "query": qtext,
                        "fields": [f"title^{title_boost}", "title.ngram^2",
                                   "body^2", "body.ngram",
                                   "chunk_text^2", "chunk_text.ngram"],
                        "type": "best_fields", "tie_breaker": 0.3}},
                    {"knn": {"vector_field": {
                        "vector": qvec, "k": 40, "boost": knn_boost}}}
                ]
            }
        }
    }

def top5(res):
    return [(h["_source"].get("doc_id"), h["_source"].get("title", "")[:40], round(h["_score"], 2))
            for h in res["hits"]["hits"][:5]]

def main():
    ap = argparse.ArgumentParser()
    ap.add_argument("--domain", required=True)
    ap.add_argument("--host", required=True)
    args = ap.parse_args()

    key = os.environ.get("OPENAI_API_KEY")
    if not key:
        print("STOP: OPENAI_API_KEY 없음 — 벡터/하이브리드 검색 불가."); sys.exit(2)

    oai = OpenAI(api_key=key)
    osc = OpenSearch(hosts=[args.host])
    index = f"aiceo-{args.domain}"
    qfile = f"poc/search/queries_{args.domain}.txt"
    queries = [q.strip() for q in open(qfile, encoding="utf-8") if q.strip()]

    # 하이브리드 boost grid (knn_boost, title_boost)
    GRID = [(4.0, 4), (8.0, 4), (8.0, 6), (12.0, 4)]

    for qt in queries:
        emb = oai.embeddings.create(model=EMBED_MODEL, input=[qt]).data[0].embedding
        assert len(emb) == EMBED_DIM
        print("\n" + "=" * 72)
        print(f"질의: {qt}")
        print("-- (a) BM25 (Nori) --")
        for r in top5(osc.search(index=index, body=q_bm25(qt))): print("   ", r)
        print("-- (b) 벡터 (knn) --")
        for r in top5(osc.search(index=index, body=q_vector(emb))): print("   ", r)
        print("-- (c) 하이브리드 (네이티브, boost grid) --")
        best = None
        for kb, tb in GRID:
            hits = top5(osc.search(index=index, body=q_hybrid(qt, emb, kb, tb)))
            top_score = hits[0][2] if hits else 0
            print(f"   [knn_boost={kb}, title_boost={tb}] top1_score={top_score}")
            for r in hits: print("      ", r)
            if best is None or top_score > best[0]:
                best = (top_score, kb, tb)
        print(f"   >> 채택: knn_boost={best[1]}, title_boost={best[2]} (top1_score={best[0]})")

if __name__ == "__main__":
    main()
```

> 하이브리드 grid의 "채택" 기준은 top1 score 최대값으로 단순화한다
> (정답지 없는 PoC라 정성 비교용). 출력에 채택한 boost 조합을 반드시
> 남긴다 — 강의에서 "왜 이 가중치인가"를 설명하기 위함.

| 7 | 컨테이너 라이프사이클 | `run-search.sh`가 기존 컨테이너 정리→기동→`until`+`curl` 헬스체크(`sleep` 미사용)→`trap` 정리를 모두 수행. 데이터 볼륨이 `poc/opensearch/data/` 하위 |
| 8 | 커밋 안전 | `git status`에 `poc/data/`·`*.jsonl`·`poc/opensearch/data/`·`.env`·임베딩/인덱스 산출물이 **추적되지 않음**(`.gitignore` 차단). 추적 대상은 코드 + `mapping.json`뿐 |
| 9 | 검색 분기 정성 확인 | 최소 1개 질의에서 BM25 top-5와 벡터 top-5가 **서로 다른 문서**를 포함(세 방식이 실제로 다르게 동작함을 눈으로 확인) |

검사 9개 중 1·2·3·4·5·6·7·8 전부 통과면 구축 성공. 9번은 정성
확인이라 "분기가 관찰됨/관찰 안 됨"을 보고에 함께 적는다(관찰 안 되면
질의 세트가 너무 쉬운 것 — §6 설계형 질의를 더 날카롭게 보강 권고).

"검색 3종이 실제로 갈렸는가" 한 줄 코멘트.
