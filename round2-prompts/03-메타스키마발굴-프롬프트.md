# 작업 지시: 도메인 데이터에 맞는 **메타 분류 스키마**를 LLM 샘플링 수렴으로 발굴

> 서울대 AI CEO 4기 · 2회차(2026-05-22) 실습용 복붙 프롬프트
> 시리즈 위치: **03 메타스키마 발굴** (← 일회성 사전 단계, 01·02의 입력을 만든다)

너는 지금 `poc/data/<domain>/` 에 **이미 수집된 도메인 검색문서(jsonl)** 가
놓여 있는 환경에 있다. 이 데이터에 **어떤 메타 분류(카테고리 체계)를 붙이면
검색·필터·부스팅에 가장 쓸모 있을지** 를, 사람이 머리로 정하지 말고
**LLM 샘플링 수렴**으로 발굴하라. 최종 산출물은
`poc/data/<domain>/meta_schema.json` **한 개**다.

이건 1회차에서 배운 "5-에이전트 병렬 수렴(같은 일을 여러 번 시켜 합의를
뽑는다)" 의 **메타 라벨 버전**이다. AI에게 데이터 구조 설계를 시키고,
여러 번의 제안을 취합해 신뢰할 수 있는 것만 채택한다.

---

## 0. 절대 규칙 (위반 시 작업 실패 — 건너뛰기 금지)

1. **입력 데이터는 이미 있다. 재수집 절대 금지.**
   `poc/data/<domain>/` 안의 검색문서 jsonl 을 그대로 쓴다. 도메인은
   `sangkwon | medical | finance | legal | policy` 중 사용자가 지정한 1개.
   - 새 크롤링·API 호출·`requests`·`curl`·웹수집 코드 작성 **금지**.
   - 입력 jsonl 이 없으면 멈추고 사용자에게 "해당 도메인 검색문서 jsonl
     부재" 를 보고한다 (없는데 만들어내지 마라).

2. **이건 런타임이 아니라 일회성 "사전 확정" 단계다.**
   메타 스키마는 검색 파이프라인이 매번 돌리는 게 아니라, **딱 한 번
   확정해서 고정**하는 설계 산출물이다. 이 작업의 결과 `meta_schema.json`
   이 곧 **02(LLM 메타 라벨링)의 읽기전용 입력**이 되고, 02가 그 스키마를
   **고정**으로 써서 모든 문서에 라벨을 붙인다. 그러므로 스키마는
   "한 번 정하면 한동안 안 바꾼다" 는 무게로 신중히 발굴한다.

3. **샘플링은 무작위 + 재현 가능해야 한다.**
   - 세트 추출 시 **난수 seed 를 고정**한다 (기본 `20260522`. 도메인의
     `_collect_meta.json` 에 `sampling` seed 가 있으면 그 값을 우선 사용).
   - 같은 입력 + 같은 seed → 같은 세트가 나와야 한다 (재현성).
   - **전수 LLM 투입 금지.** 문서가 수백·수천 건이어도 전부 LLM 에 넣지
     않는다 (비용·시간). 오직 **샘플 세트만** LLM 에 던진다.
   - 세트 크기·개수는 **데이터 양에 맞춰 정한다** (§3-2 표 참조).
     예: 문서 ~300건 → 10건 × 10세트. 문서 ~3,000건 → 30건 × 10세트.

4. **수렴 판정 임계를 명시하고 그대로 적용한다.**
   - N 개 세트의 LLM 제안을 취합한 뒤, **N 의 60% 이상 세트에서 등장한
     카테고리만 "채택(confirmed)"** 한다.
   - 30%~60% 세트에서 등장 → **"보류(candidate)"** 로 분리 기록 (채택 안 함).
   - 30% 미만(특히 단일 세트에만 등장) → **"기각(rejected)"**, 스키마에서 제외.
   - 임계 퍼센트와 각 카테고리의 등장 세트 수를 `meta_schema.json` 의
     `_provenance` 에 반드시 남긴다 (왜 채택했는지 추적 가능해야 함).

5. **사내 기본값(커뮤니티 12개 mid_category)을 베끼지 마라.**
   메디게이트 커뮤니티는 `mid_category` 12개 선택지
   (`정치/행정/법률, 경제/산업/노동, 사회/인권/젠더, 보건/의료/복지,
   국제/외교/안보, 과학/기술/환경, 교육/문화/종교, 미디어/언론,
   연예/스포츠, 부동산/재테크/투자, 라이프스타일/소비, 기타`) 를 쓴다.
   - 이건 **커뮤니티 게시글** 도메인에서 수렴해 나온 결과지, 정답표가 아니다.
   - 지금 다루는 도메인이 법령·약가·판례·예산이면 **다른 체계가 나오는 게
     정상**이다 (예: 법령 → `소관부처 / 법령분야 / 규율대상`). 무리해서
     12개에 끼워 맞추지 마라. 12개는 **"비교 기준선"** 으로만 본다
     (수렴 결과가 그것과 얼마나 다른지 §5 에서 대조 보고).

6. **LLM API 키가 없으면 멈추고 보고한다 (임의 진행 금지).**
   `OPENAI_API_KEY` / `GEMINI_API_KEY` / `ANTHROPIC_API_KEY` 중
   사용 가능한 키가 하나도 없으면, **샘플 세트만 추출**한 뒤
   "LLM 키 부재로 발굴 LLM 호출 불가 — 세트는 추출 완료, 키 주입 후
   재실행 필요" 를 사용자에게 보고하고 종료한다. 키 없이 추측으로
   카테고리를 지어내지 마라.
   - 그 외의 모호함에는 사용자에게 질문하지 마라. 이 명세는 완결돼 있다.

7. **산출물·코퍼스 노출 경계를 지킨다.**
   - `meta_schema.json` 은 **스키마 정의**(카테고리 이름·필드 구조)이므로
     공개 가능 — 제3자 본문이 들어가면 안 된다. `_provenance` 의
     "근거 예시" 에도 **원문 본문을 통째로 박지 말고**, doc_id 와
     15자 이내 라벨 근거만 남긴다.
   - 발굴에 쓴 **샘플 코퍼스 자체**(추출한 jsonl 세트)는 제3자 저작권
     본문이다. `poc/data/` 전체와 `*.jsonl` 은 이미 `.gitignore` 로
     커밋 차단돼 있음을 작업 시작 시 1회 확인하고 보고한다
     (확인만; .gitignore 수정 금지).

## 1. 무엇을 만드나 (산출물 = 파일 1개)

| # | 경로 | 내용 |
|---|---|---|
| 1 | `poc/data/<domain>/meta_schema.json` | 발굴·수렴된 **메타 분류 스키마** + 발굴 근거(`_provenance`) |

`<domain>` = 사용자 지정 도메인 디렉터리명 (`sangkwon`/`medical`/`finance`/`legal`/`policy`).

### 1-1. 왜 이게 필요한가 (CEO 관점 30초 설명)

검색을 잘 하려면 "이 문서가 무엇에 관한 것인지" 를 기계가 알아야 한다.
사람이 카테고리 표를 손으로 짜면 (a) 데이터 실제 분포와 안 맞고
(b) 짜는 사람 편향이 들어간다. 그래서 **AI 에게 "이 문서들 잘 나누는
분류 체계를 제안해 봐" 를 여러 번 시키고, 여러 번 공통으로 나온 것만
믿는다.** 사람은 최종 검수만 한다. → "AI 로 데이터 구조를 발견하는 법".

### 1-2. 메타 스키마가 실제로 어떤 모양인지 (참조: 운영 커뮤니티 스키마)

메디게이트 커뮤니티 운영 인덱스는 LLM 이 게시글마다 아래를 생성한다
(이게 "메타 스키마의 실제 구조" 의 레퍼런스다 — 우리가 발굴할 것도
같은 모양의 **계층형 카테고리 + 보조 필드** 다):

- **1단계 기본 분류**(GPT-4o-mini 1회 호출):
  `main_category`(대분류, keyword) → `mid_category`(중분류, 12개 선택지)
  → `sub_category`(소분류, 15자 이내) + `description`(1줄 요약) +
  `keywords`(3~5개) + `system_alert`(boolean) 등.
- **2단계 상세 분류**(Gemini + GPT 병렬 → 합산): `Type` +
  `Topics[]`·`Keywords[]`·도메인 엔티티 배열(질병/약물/진료과 …)
  각각 `관련성` 0.0~1.0.
- 핵심 패턴: **계층(main→mid→sub)** + **고정 선택지(enum)** +
  **자유 텍스트 보조필드(description/keywords)** + **관련성 점수**.
  → 우리가 발굴할 `meta_schema.json` 도 정확히 이 골격을 따른다.
  단, **선택지 내용은 도메인 데이터에서 수렴시켜 새로 채운다.**

> 커뮤니티의 `mid_category` 12개도 원래는 누군가 "여러 게시글을 보고
> 분류 체계를 합의" 한 결과다. 우리는 그 합의 과정을 **LLM 샘플링 수렴**
> 으로 자동화·재현하는 것이다.

## 2. 수행 절차 (이 순서대로 — 건너뛰기 금지)

```
[A] 환경·입력 점검 (도메인 확정, jsonl 존재, .gitignore 차단, LLM 키)
        │
[B] 세트 추출  (seed 고정 무작위 → N개 세트, 각 세트 K건)
        │
[C] 세트별 LLM 발굴  (각 세트를 §4 프롬프트로 LLM 에 던져 카테고리 체계 제안)
        │
[D] 취합 → 수렴 판정  (등장 세트 수 집계 → 60%+ 채택 / 30~60% 보류 / 30%- 기각)
        │
[E] meta_schema.json 산출  (확정 스키마 + _provenance)
        │
[F] 사내 12개 기준선 대조 + 자가검증 + 보고
```

### Step A — 환경·입력 점검

1. 사용자가 지정한 도메인을 확인한다 (`sangkwon`/`medical`/`finance`/`legal`/`policy`).
   미지정이면 §0-6 예외가 아니므로 **여기서만** 1회 질문 허용:
   "어느 도메인의 메타 스키마를 발굴할까요? (5개 중 1개)".
2. `poc/data/<domain>/` 의 **검색문서 jsonl** 을 식별한다. 도메인별 기본:

   | domain | 검색문서 jsonl (기본) | 비고 |
   |---|---|---|
   | `sangkwon` | `policy_news.jsonl` | 상권/소상공인 정책뉴스 |
   | `medical` | `drug_detail.jsonl` | 의약품 허가상세 본문 |
   | `finance` | `policy_news.jsonl` | 금융/연금 정책뉴스 |
   | `legal` | `law_corpus.jsonl` (주), `precedent_refs.jsonl` (보조) | 법령 본문 |
   | `policy` | `press_release.jsonl` | 정부 보도자료 |

   - 디렉터리에 jsonl 이 위 이름과 다르면 **`*.jsonl` 중 `body`(또는
     본문 성격 필드)가 있는 가장 큰 파일** 을 검색문서로 택한다.
   - `_collect_meta.json` 이 있으면 읽어 `count`(문서 수)·`schema`·
     `sampling`(seed) 를 파악한다. seed 가 거기 있으면 그 값을 §3-3 에 쓴다.
3. `.gitignore` 에서 `poc/data/` 와 `*.jsonl` 차단 라인 존재를 grep 으로
   1회 확인하고 결과를 보고한다 (수정 금지, 확인만).
4. LLM 키 확인: `OPENAI_API_KEY`/`GEMINI_API_KEY`/`ANTHROPIC_API_KEY`.
   하나도 없으면 Step B 까지만 하고 §0-6 대로 보고 후 종료.

### Step B — 세트 추출 (무작위·재현가능)

§3 의 규칙대로 jsonl 에서 **N 개 세트**를 뽑는다. 각 세트는 서로 겹쳐도
무방하나(독립 표본), seed 고정으로 **재현 가능**해야 한다. 각 세트의
각 문서는 LLM 에 넣을 때 **`title` + `body` 앞부분(문서당 최대 1,500자)**
만 사용한다 (긴 본문 통째 투입 금지 — 토큰·비용).

### Step C — 세트별 LLM 발굴 (병렬 권장)

N 개 세트 각각을 §4 의 **발굴 LLM 프롬프트**로 LLM 에 1회씩 던진다.
세트 간 독립이므로 병렬 호출 가능. LLM 응답은 §4 가 강제하는 JSON
스키마로 받는다. 호출 실패한 세트는 최대 2회 재시도, 3회째 실패면
그 세트를 "결측" 으로 기록하고 **남은 세트로 수렴 계속**(전체 중단 금지,
단 유효 세트가 N 의 절반 미만이면 사용자에게 보고).

### Step D — 취합 → 수렴 판정

1. 모든 세트 응답에서 제안된 카테고리를 **계층 경로 단위로 정규화**한다
   (`main/mid` 또는 `main/mid/sub` 의 표준화된 한국어 명칭. 동의어·표기차
   는 LLM 또는 규칙으로 병합: 예 "보건의료" ≈ "보건/의료").
2. 각 정규화 카테고리가 **몇 개 세트에서 등장**했는지 카운트한다.
3. §0-4 임계 적용:
   - 등장 세트 수 ≥ ceil(N × 0.60) → **confirmed** (스키마에 채택)
   - ceil(N × 0.30) ≤ 등장 < ceil(N × 0.60) → **candidate** (보류, 별도 기록)
   - 등장 < ceil(N × 0.30) → **rejected** (제외)
4. `description`·`keywords` 등 **보조 필드의 적정성**도 같은 방식으로
   수렴: 다수 세트가 "1줄 요약 필요/키워드 3~5개 필요" 라고 제안하면 채택.
5. 계층 깊이(2단 vs 3단)도 수렴 결과로 결정한다. sub_category 가
   confirmed 가 거의 없으면 2단(main/mid)으로 확정한다 (억지 3단 금지).

### Step E — `meta_schema.json` 산출

§6 의 JSON 형태로 **확정 스키마 + `_provenance`** 를 쓴다.
경로: `poc/data/<domain>/meta_schema.json` (정확히 이 위치 — 02 가
이 경로를 읽기전용으로 로드).

### Step F — 기준선 대조 + 자가검증 + 보고

§5·§7 수행 후 사용자에게 표로 보고하고 종료.

## 3. 샘플링 규칙 (재현성의 핵심)

### 3-1. 원칙

- **무작위 추출 + seed 고정**: 같은 jsonl·같은 seed → 같은 세트.
- **세트 단위**: "K 건짜리 묶음" 을 한 세트로, 총 N 세트. 각 세트를
  독립적으로 LLM 에 던져 **독립 의견 N 개**를 만든다 (이게 수렴의 재료).
- **세트 간 중복 허용**: 세트마다 새로 무작위 추출(복원추출 가능).
  중요한 건 "독립 표본 N 개" 지 "전수 커버" 가 아니다.

### 3-2. 데이터 양에 맞춘 세트 크기·개수 (가이드)

| 검색문서 총 건수 | 세트 크기 K | 세트 수 N | LLM 호출 = N회 |
|---|---|---|---|
| ~100건 미만 | 10 | 8 | 8 |
| 100 ~ 500건 | 10 | 10 | 10 |
| 500 ~ 2,000건 | 30 | 10 | 10 |
| 2,000 ~ 10,000건 | 50 | 10 | 10 |
| 10,000건 이상 | 100 | 10 | 10 |

- N 은 **수렴 신뢰의 분모**다. 최소 8 이상 유지 (8 미만이면 60% 임계가
  의미 약함). 비용 상한 때문에 N 을 줄여야 하면 사용자에게 보고.
- K 가 너무 작으면(<10) 세트 1개가 도메인을 못 대표한다. 최소 10.
- 표 경계는 가이드다. `_collect_meta.json` 의 `count` 로 실제 건수를
  확인해 한 칸 안에서 택하되, **선택한 K·N 을 `_provenance` 에 기록**한다.

### 3-3. seed 결정 순서

1. `poc/data/<domain>/_collect_meta.json` 의 `sampling` 에 seed 가
   명시돼 있으면 **그 값** (예: legal = `20260522`).
2. 없으면 **기본 `20260522`** (강의일자 — 전 도메인 공통 재현 기준).
3. 선택한 seed 를 `meta_schema.json._provenance.sampling.seed` 에 기록.

### 3-4. 세트 추출 의사코드 (그대로 구현)

```python
import json, random

random.seed(SEED)                       # §3-3 으로 결정한 정수
docs = [json.loads(l) for l in open(JSONL_PATH, encoding="utf-8")]
sets = []
for s in range(N):                      # N = §3-2
    picked = random.sample(docs, k=min(K, len(docs)))   # 세트 내 비복원
    # 세트 간에는 매번 새로 sample → 세트 간 중복 허용(독립 표본)
    payload = [{
        "doc_id": d.get("doc_id"),
        "title": d.get("title", "")[:200],
        "body":  (d.get("body") or d.get("content") or "")[:1500],
    } for d in picked]
    sets.append({"set_index": s, "docs": payload})
```

> `random.sample` 은 seed 고정 시 결정적이다. 재현성 확보의 핵심.

## 4. 발굴 LLM 프롬프트 (각 세트를 던질 때 **이 프롬프트를 그대로** 사용)

세트 1개당 LLM 1회 호출. system + user 를 아래대로 구성한다.
LLM 이 추측·자유서술 못 하게 **출력 JSON 스키마를 강제**한다.

### 4-1. system 프롬프트 (고정)

```
당신은 검색 시스템 설계자다. 주어진 도메인 문서 묶음을 보고, 이 문서들을
'검색·필터·랭킹 부스팅' 에 가장 쓸모 있게 나눌 메타 분류 체계를 설계한다.

규칙:
- 분류는 계층형이다: main_category(대분류) → mid_category(중분류) →
  sub_category(소분류, 선택적). 실제 문서에서 변별력이 있을 때만 sub 를 둔다.
- 카테고리 이름은 한국어, 짧고(중분류 12자·소분류 15자 이내) 상호배타적으로.
- 이 묶음에 실제로 나타난 주제만 근거로 삼는다. 일반론으로 채우지 마라.
- 보조 필드(description=1줄요약, keywords=3~5개)가 이 도메인 검색에
  유용한지도 판단해 권고한다.
- 출력은 아래 JSON 스키마만. 설명 문장·마크다운·코드펜스 금지.
```

### 4-2. user 프롬프트 (세트 주입)

```
[도메인]: <domain>  (예: legal=한국 법령 본문 / medical=의약품 허가상세 등)

[이 묶음의 문서 N건] (title + body 발췌):
<세트의 docs[] 를 아래 형식으로 나열>
- doc_id: <id>
  title: <title>
  body: <body 1500자 발췌>
(… K건 반복 …)

[요청]
1. 이 문서들을 잘 분류하는 main_category 후보(2~6개)와, 각 main 아래
   mid_category 후보(각 2~8개)를 제안하라. 변별력 있으면 sub_category 도.
2. 각 카테고리에 'description'(이 카테고리가 뭘 담는지 1줄)과
   'example_doc_ids'(이 묶음에서 그 카테고리에 해당하는 doc_id 1~3개)를 달라.
3. 이 도메인 검색에서 문서별 보조 필드로 'description'(1줄요약)과
   'keywords'(3~5개)를 두는 게 유용한지 yes/no 와 한 줄 이유.
4. 아래 JSON 으로만 답하라:

{
  "main_categories": [
    {
      "name": "<대분류>",
      "description": "<1줄>",
      "mids": [
        {
          "name": "<중분류>",
          "description": "<1줄>",
          "subs": ["<소분류>", ...],          // 없으면 []
          "example_doc_ids": ["<id>", ...]
        }
      ]
    }
  ],
  "aux_fields_recommended": {
    "description": {"useful": true, "reason": "<1줄>"},
    "keywords":    {"useful": true, "reason": "<1줄>"}
  }
}
```

> **중요**: LLM 에 "12개 커뮤니티 카테고리" 를 절대 프롬프트에 노출하지
> 마라. 노출하면 LLM 이 그걸 베껴 수렴이 오염된다. 도메인 데이터에서
> **백지 상태로** 떠오른 체계를 받아야 진짜 수렴이다. 12개와의 대조는
> 발굴이 끝난 뒤 §5 에서 사람(에이전트)이 한다.

## 5. 사내 기본값(커뮤니티 12개) 대비 — 발굴 후 대조 (LLM 에 노출 금지, 보고용)

수렴 완료 후, 확정된 mid_category 집합을 커뮤니티 12개
(`정치/행정/법률, 경제/산업/노동, 사회/인권/젠더, 보건/의료/복지,
국제/외교/안보, 과학/기술/환경, 교육/문화/종교, 미디어/언론,
연예/스포츠, 부동산/재테크/투자, 라이프스타일/소비, 기타`) 와 대조해
다음을 보고한다 (스키마에는 넣지 않고 **보고 메시지에만**):

- **겹치는 것**: 발굴 카테고리 중 12개와 의미가 같은 것 (예 `보건/의료/복지`).
- **도메인 고유로 새로 나온 것**: 12개에 없는 것 (예 법령 → `소관부처별`,
  약가 → `제형/투여경로`, 판례 → `사건유형(민사/형사)`).
- **판정 문구**: "도메인 특성상 사내 12개와 N개 일치 / M개는 도메인
  고유 — 12개를 그대로 베끼지 않고 데이터에서 수렴한 결과" 를 명시.
  (도메인이 법령·약가류면 12개와 거의 안 겹치는 게 **정상**이다.)

## 6. `meta_schema.json` 산출 형식 (이 구조로 Write)

아래는 **legal 도메인 예시**다. 실제 값은 너의 수렴 결과로 채운다.
구조(키·중첩·`_provenance`)는 그대로 따른다.

```json
{
  "domain": "legal",
  "schema_version": "1.0",
  "generated_at": "2026-05-22T00:00:00+09:00",
  "source_corpus": {
    "file": "law_corpus.jsonl",
    "doc_count": 449,
```

- `categories.*.enum` 에는 **confirmed 만** 넣는다. candidate 는
  `_provenance.category_evidence` 에 `status:"candidate"` 로만 남기고
  enum 에는 절대 넣지 않는다 (보류는 보류).
- `hierarchy_depth` 는 수렴 결과에 맞춰 2 또는 3. sub 가 enum 으로
  안 굳을 정도면 `sub_category.free=true` (LLM 생성 자유필드)로 둔다.
- 원천 jsonl 에 **이미 구조화 필드**(예 legal `ministry`, policy
  `ministry`)가 있으면 라벨링 대상이 아니라 그대로 패싯 가능 →
  `aux_fields.<field>.from_source_field` 로 명시 (LLM 라벨 낭비 방지).
- `_provenance` 의 `example_doc_ids` 는 **doc_id 만**. 본문 발췌 금지(§0-7).

|---|---|
| (1) 파일 존재 | `poc/data/<domain>/meta_schema.json` 존재 |
| (2) JSON 유효 | 파싱 OK |
| (3) 필수 키 | 최상위 + `_provenance` 필수 키 전부 존재 |
| (4) enum 순도 | candidate 가 enum 에 누수 0건 |

| (6) 재현성 | seed·K·N 이 `_provenance` 에 기록됨 |

전부 통과면 성공. 사용자에게 아래를 표로 보고하고 종료:
- 사용 도메인 / 검색문서 jsonl / 문서 수
- 샘플링 (seed, K, N) — 재현 가능 명시
- 수렴 결과: confirmed 카테고리 수 / candidate 수 / rejected 수
- 사내 12개 대비 (overlap / domain-specific / verdict)
- 사용 LLM (provider·model) — 키 없었으면 "세트만 추출, 키 부재로 발굴 미수행"
- 6개 자가검증 결과 표

LLM 키 부재로 발굴을 못 한 경우: meta_schema.json 을 **쓰지 말고**

보고하고 종료한다 (빈 스키마·추측 스키마 생성 금지).

## 시리즈 연결 (이 산출물이 어디로 흐르나)

이 작업은 **검색 시리즈의 일회성 사전 단계**다:

```
03 (이 작업) 메타스키마 발굴
  └─ poc/data/<domain>/meta_schema.json  ← 한 번 확정하면 고정
        │  (읽기전용 로드)
        ▼
02 LLM 메타 라벨링
  └─ 위 meta_schema.json 의 enum·필드를 고정 사용해, 각 검색문서에
     main/mid/sub_category + description + keywords 를 LLM 으로 라벨링
        │
        ▼
01 검색 인덱싱
  └─ 라벨된 메타를 인덱스 필드로 색인 → 질의 시 그 메타로
     하이브리드 부스팅(키워드·메타 필터·벡터 결합)
```

요약: **03 이 "분류 체계(틀)" 를 데이터에서 발굴 → 02 가 그 틀로 모든
문서에 "라벨" 을 붙임 → 01 이 그 라벨로 "검색 부스팅"**. 그래서 03 의
스키마가 흔들리면 02·01 전체가 흔들린다 — §0-2 대로 신중히 확정한다.

이건 1회차 "같은 일을 여러 에이전트에 시켜 합의를 뽑는다" 의
**메타 라벨 버전**이다. CEO 핵심 교훈: **데이터 구조(택소노미)조차
사람이 손으로 정하지 않고, AI 다중 샘플의 수렴으로 발견할 수 있다.**
