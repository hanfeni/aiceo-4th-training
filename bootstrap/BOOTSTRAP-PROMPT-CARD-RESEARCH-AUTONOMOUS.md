# 작업 지시: `/card-research` 명함 기반 인물 리서치 하네스 자율 구축

너는 지금 이 하네스가 **설치되지 않은** 환경에 있다. 아래는 이 하네스가
**만족해야 하는 설계 명세**다. 원문 파일은 주어지지 않는다. **명세를
이해하고 네가 직접 파일을 작성해** 하네스를 구축하라.

받아쓰기가 아니다. 표현·문장·구조는 네가 정한다. 단 아래 **기능 계약·
불변식·결함 처방·윤리 경계**는 한 줄도 빠뜨리지 말고 구현하라. 글자가
아니라 **계약을 지키는 것**이 합격선이다.

작업 중 사용자에게 질문하지 마라(명세가 완결돼 있다). `pwd`로 현재
프로젝트 루트를 확인하고 그 기준으로 경로를 정한다. 이 단계에서
하네스를 실제로 실행하지는 마라(구축까지만).

---

## 1. 목적 (WHY)

사용자가 **사람 명함 정보**(이름 + 회사/소속 + 직책 + 가용 시 연락처·
도메인·이미지/텍스트)를 입력하면, 그 인물과 소속 조직의 **공개 정보를
웹서치로 리서치해 "살을 붙인" 인물 파일 1개**를 산출한다.

**정체성 원칙 (반드시 구현):**

- **도메인 무관.** 대상은 의학 전문가에 한정하지 않는다. 의사·교수·
  기업인·공무원·예술가·일반 직장인 등 누구든 받는다. 하네스는
  직업군을 **먼저 판별**해 정보 수집 축을 그 직업에 맞게 **동적으로**
  정한다. 특정 도메인(의료 등) 고정 프리셋을 박지 마라.
- **명함은 신원 앵커(identity anchor)다.** 명함의 이름+회사+직책+
  도메인이 "이 사람"을 고정하는 기준점이다. 검색으로 찾은 정보가
  이 앵커와 **불일치하면 동일인으로 단정하지 말 것**(동명이의 방어 —
  §6 핵심). 환각보다 신원 오인이 더 치명적이다.
- **공개 직업 정보만.** 수집 경계는 §4. 사생활은 수집하지 않는다.
- 리서치는 **모아 정리하는 것**이지 인물을 평가·등급화하지 않는다
  (단 공개된 평판·논란이 출처와 함께 있으면 사실로 기록은 가능).

명함 입력이 비어 있으면 명함 정보를 한 번만 물어 받고, 그 외에는
질문 없이 끝까지 수행한다.

## 2. 만들 파일과 책임 (WHAT — 구현은 네 몫)

`.claude/` 하위에 커맨드 1 + 에이전트 3 + 스킬 1 = **5파일**. 경로:

| 파일 | 역할 (이 책임을 네 표현으로 구현) |
|---|---|
| `.claude/commands/card-research.md` | 오케스트레이터. 공유 메타 산출 + 셀렉터 호출 + 정보축 N병렬 발사 + 검증 + 신원파일 조립 + md 1개 저장 |
| `.claude/agents/card-parser.md` | 셀렉터. 명함(이미지/텍스트)에서 이름·회사·직책·연락처·도메인 추출 + **직업군 판별** + 정보 수집 축 N개 동적 결정 + 신원 앵커 확정 |
| `.claude/agents/card-researcher.md` | 리서치 일꾼. 배정된 **정보 축 1개**만 공개 웹서치로 수집, 앵커 대조, 압축 결과만 반환(분장) |
| `.claude/agents/card-validator.md` | 검증자. 동명이의·환각(URL/인용 실재)·PII 경계 위반·앵커 불일치를 검증, REMOVE/플래그 |
| `.claude/skills/card-research/SKILL.md` | 자연어 자동인식 진입점. 커맨드 실행순서로 연결 |

researcher 는 N개를 **같은 정의 1개**(`card-researcher`)로 호출한다
(축마다 별도 파일 아님 — 동일 작업이라 input 의 `axis` 만 다름).
parser 가 정한 축 수만큼 오케스트레이터가 병렬 호출한다.

에이전트 정의는 Claude Code subagent 형식(YAML frontmatter `name`/
`description`/`tools`/`model` + 본문). `tools` 는 책임에 필요한 최소로
네가 판단(researcher 는 WebSearch/WebFetch, parser 는 명함이 이미지면
Read 포함, validator 는 WebFetch + raw 정리에 필요한 것).

## 3. 토폴로지 (순서·병렬성 고정, 서술은 네 몫)

```
Step 0  오케스트레이터: 공유 메타 산출 + 저장 경로 결정          [본체]
Step 1  card-parser ×1   [순차 선행 · 명함→앵커+직업군+정보축 N개]
Step 2  card-researcher ×N  [축별 분장 병렬 · 단일 메시지 N개 동시]
Step 3  card-validator ×N   [병렬 · 동명이의/환각/PII 검증 · raw 정리]
Step 4  오케스트레이터: 검증 통과분으로 신원 파일 조립
Step 5  최종 md 1개 저장 + 작업 로그 + 보고
```

- **Step 0 공유 메타**: `now_kst`, `today`, `run_id`, `subject_slug`
  (인물명+회사 kebab-case). 한 번만 산출해 전 에이전트에 동일 주입.
  어느 에이전트도 자체 `date` 호출 안 함.
- **Step 1 parser (셀렉터)**: 명함에서 필드 추출 → **신원 앵커**
  확정(이름·회사·직책·도메인·연락처 중 가용한 것) → 그 인물의
  **직업군 판별**(WebSearch 1~2회로 "이 회사·직책이 무슨 분야인지"
  확인) → 직업군에 맞는 **정보 수집 축 2~6개**를 동적으로 결정.
  축 예(직업군마다 다르게): 소속·재직 이력 / 경력·학력 / 저작·
  논문·특허·작품 / 공개 발표·인터뷰·미디어 / 공개 전문 SNS /
  소속 조직 개요. **의료 등 특정 도메인 고정 금지** — 직업군에
  맞춰 축 자체를 다르게 정한다(기업인≠예술가≠연구자).
- **Step 2 researcher (분장 병렬)**: parser 가 정한 축마다 1개씩
  **단일 메시지로 동시** 호출. 각 researcher 는 배정된 **축 1개만**
  공개 웹서치. 모든 항목을 신원 앵커와 대조해 `anchor_match:
  yes|maybe|no` 부착. 압축 결과만 반환(불변식 2).
- **Step 3 validator (병렬)**: 축별 raw 를 검증 — URL 실재
  (WebFetch 표본), 인용 무결성, **동명이의**(anchor_match=no 는
  기본 폐기, maybe 는 "미확정"으로 분리), **PII 경계 위반**(사생활
  정보면 REMOVE). raw=검증 통과분 불변식 유지.
- **Step 4 조립**: 검증 통과 항목을 신원 앵커 중심으로 **인물
  파일**로 조립(KYC식 엔티티 파일). 미확정(maybe)·정보 없음 축은
  정직하게 그 상태로 표기.
- **Step 5 저장**: md 1개. 그 외 md 안 만든다(researcher 는 raw 를
  파일로 쓰되 — §5 불변식 2 — 최종 산출 md 는 1개).

## 4. PII 윤리 경계 (이 하네스의 헌법 — 위반 시 실패)

수집·기록 **허용**:
- 공개된 직업적 활동: 소속·재직 이력, 직책, 학력·경력, 저작·논문·
  특허·작품·수상, 공개 발표·강연·인터뷰·미디어 출연, 소속 조직의
  공개 개요, 본인이 **직업 목적으로 공개한** SNS·블로그·홈페이지
  (예: LinkedIn 공개 프로필, 회사 소개 페이지, 공개 학술 프로필).

수집·기록 **금지** (발견해도 파일에 쓰지 않는다):
- 사생활: 가족·혼인·자녀, 자택 주소·개인 휴대폰·주민번호 등 식별
  민감정보, 건강·종교·정치성향·성적지향, 개인 사생활 SNS 계정·
  사진, 미성년 관련 정보, 제3자가 유출·폭로한 비공개 정보.
- 경계가 모호하면 **보수적으로 제외**하고 그 사실을 산출물에
  "수집 경계로 제외함"으로 1줄 남긴다(왜 비었는지 정직).

이 경계는 parser·researcher·validator 세 곳 모두에 명시한다.
researcher 는 금지 정보를 애초에 수집하지 않고, validator 는
혹시 들어온 금지 정보를 REMOVE 하는 2중 방어.

## 5. 불변식 (어기면 하네스 실패)

1. **공유 메타·앵커 동일 주입**: 신원 앵커(명함에서 확정)는 parser
   가 1회 산출해 전 researcher·validator 에 **글자 단위 동일** 주입.
   researcher 가 앵커를 변형·추측하지 않는다.
2. **에이전트 반환은 압축**: researcher 는 raw 를 파일로 직접 Write
   하고(raw_dir), 오케스트레이터에는 **압축 결과 목록만** 반환:
   항목당 `{축, 사실 1줄(≤200자), url, 출처유형, anchor_match,
   posted_at?}`. 장문 본문·페이지 덤프 금지(컨텍스트 폭주 방지).
3. **단일 조립 지점**: 신원 파일 조립은 오케스트레이터 1곳에서만.
   researcher 는 자기 축만, 서로 합치지 않는다.
4. **신원 앵커 대조 강제**: 모든 수집 항목은 `anchor_match`
   (yes/maybe/no)를 가져야 한다. 없는 항목은 폐기. `no`는 기본
   제외, `maybe`는 "동일인 미확정" 섹션으로 분리(본문에 섞지 마라).
5. **결정적 단계 자가검증**: 최종 md 저장 후, 파일 존재 + 모든
   사실 항목에 url 이 있는지 + 금지 PII 패턴이 본문에 없는지를
   `grep` 등으로 **수행 후 기계 확인**한 뒤에만 "저장 완료" 보고
   (확인 없이 했다고 가정·허위 기록 금지).

## 6. 결함 처방 (이미 비싸게 학습된 함정 — 회피 설계 필수)

아래는 유사 멀티에이전트 하네스를 여러 번 돌려 얻은 실패의 **일반
원리**다. 구체 사례·고유명은 의도적으로 뺐다(처방만 구현하라).

- **동명이의·신원 오인 (이 하네스 1순위 위험)**: 같은 이름의 다른
  사람 정보를 그 사람 것으로 붙이면 환각보다 치명적이다. 처방:
  명함을 신원 앵커로 고정 → 모든 항목에 `anchor_match` 강제 →
  회사·직책·도메인 중 **2개 이상 일치**해야 `yes`, 1개만이면
  `maybe`, 상충하면 `no`. `no` 제외, `maybe` 분리. 흔한 이름은
  특히 보수적으로.
- **환각 방어**: url·출처유형·사실문 3종 없는 항목 폐기. 검색
  결과에 없는 경력·직함을 지어내지 않는다. 정보 없으면 "공개
  정보 없음"으로 정직히(거짓 채움 금지).
- **0건의 2종 구별 (정보 없는 사람 ≠ 검색 부실)**: 어떤 축이
  비었을 때 — `empty_searched`(쿼리를 충분히 변형·실행했으나
  공개 정보가 실제로 없음 = 정직한 0건, 정상) vs
  `under_searched`(시도 부족 = 허용 안 됨). researcher 는 축마다
  최소 다중 쿼리 변형을 실행하고 `searched_queries`를 raw 에
  증거로 남긴다. 도메인 무관이라 일반인은 진짜 정보가 거의 없을
  수 있다 — 그건 정직한 empty_searched 로 보고하고 지어내지 마라.
- **B. 컨텍스트 폭주 회피**: researcher 결과가 오케스트레이터로
  모이면 출력이 잘린다(예: 50K 절단). 반환을 압축 결과 목록으로
  강제(불변식 2). raw 본문은 파일에만.
- **PII 경계 침범 회피**: §4 를 parser·researcher·validator 3곳에
  명시. 모호하면 제외하고 "경계로 제외함" 1줄. 사생활 폭로성
  검색 쿼리를 애초에 만들지 마라.
- **F. 비결정 단계 누락 회피**: 저장·집계 같은 결정적 단계는
  "하라"만 적지 말고 수행 후 grep 등으로 재확인하게 한다(불변식 5).
- **G. 이식 이력 누출 회피**: 네가 작성하는 5파일에 개발 이력·
  고유 인물 사례명·"재구축/복제" 같은 표현을 넣지 마라. 규칙의
  **정당성 근거**는 적되 "언제 누가 발견했는가"는 적지 않는다.
- **도메인 고정 회피 (이 하네스 고유)**: parser 가 정보 축을
  의료 등 특정 분야로 고정하면 다른 직업군에서 빈 결과가 된다.
  축은 **판별된 직업군에 맞춰 동적으로** 정하도록 명시한다.

## 7. 산출물 Contract (최종 md 1개 — 구조 고정, 표현은 네 몫)

저장 경로: `./specs/card-research/<subject_slug>/<today>/00_profile.md`
(같은 날 재실행 시 `00_profile_<run_id>.md` 누적 — 덮어쓰기 금지).
프로젝트 로컬, 전역 vault 금지.

최상단 YAML frontmatter 에 **다음 키를 전부** 포함(이름·구조 고정,
값은 실행마다 다름. Contract 외 키 추가 금지):

```
harness: card-research
subject_name: <명함의 이름>
subject_org: <명함의 회사/소속>
subject_title: <명함의 직책>
subject_slug: <subject_slug>
run_at: <now_kst>
run_id: <run_id>
occupation_class: <parser가 판별한 직업군>
axes_run: <리서치한 정보 축 수>
total_facts: <검증 통과한 사실 항목 수>
anchor_dist: { "yes": N, "maybe": N }
axes_empty: <공개정보 없어 비운 축 수>
pii_excluded: <PII 경계로 제외한 건수>
```

본문 필수 섹션(순서 자유, 5개 모두):

1. **인물 요약** — 명함 앵커(이름·소속·직책) + 판별 직업군 +
   확정(anchor yes) 사실 핵심을 2~4문장. 신원이 확실한 것만.
2. **확정 프로필** — anchor_match=yes 사실을 축별로 정리. 각 항목
   `사실 | 출처 url | 출처유형`. 직업군에 맞는 축 구성(고정 아님).
3. **동일인 미확정** — anchor_match=maybe 항목(같은 이름이나 신원
   확증 안 됨). "확정 아님" 명시. no 는 싣지 않음.
4. **소속 조직 개요** — 명함 회사의 공개 정보(사업·규모·대표 등
   공개 범위). 없으면 "공개 정보 없음".
5. **수집 메모** — 비운 축과 사유(empty_searched/정보없음),
   PII 경계로 제외한 건수, 동명이의 처리 요약, 한계. 정직하게.

인물 평가·등급·추측은 본문에 쓰지 않는다(공개된 사실·평판은
출처와 함께 기록 가능, 하네스의 주관 판정은 금지).

## 8. 수행 절차

1. `pwd` 로 루트 확인. `.claude/{commands,agents,skills/card-research}`
   와 `.claude/history/daily` 디렉터리 생성.
2. §1~7 을 충족하는 **5개 파일을 네 표현으로 작성**해 각 경로에 Write.
   원문을 받지 않았으므로 베낄 것이 없다. 명세를 구현하라. 각
   에이전트 정의 말미에 "출력 직전 자가검증 체크리스트"를 네가
   설계해 넣어라(특히 anchor·PII·환각).
3. §9 자가검증 수행, 결과 표 보고.
4. 하네스를 실제로 실행하지 마라(구축까지만).

## 9. 자가검증 (기능 계약 충족도 — 글자가 아니라 계약)

1:1 비교는 하지 않는다(자율 작성이라 글자가 달라야 정상). 아래를
점검하고 PASS/FAIL 보고. FAIL 이면 그 파일을 계약에 맞게 고친 뒤
재점검한다.

```bash
echo "=== (1) 5파일 존재 ==="
for f in .claude/commands/card-research.md \
  .claude/agents/card-parser.md \
  .claude/agents/card-researcher.md \
  .claude/agents/card-validator.md \
  .claude/skills/card-research/SKILL.md ; do
  [ -f "$f" ] && echo "OK $f" || echo "FAIL $f MISSING"
done

echo "=== (2) 각 파일이 자기 책임 개념을 담는가 (개념 존재 — 글자 아님) ==="
grep -lE "셀렉터|N병렬|분장|신원 앵커|조립|md 1개" .claude/commands/card-research.md && echo "  cmd: 오케스트레이션 개념 O"
grep -lE "직업군|판별|정보 축|앵커 확정|동적" .claude/agents/card-parser.md && echo "  parser: 직업군판별+동적축+앵커 O"
grep -lE "축 1개|분장|압축 결과|anchor_match|반환" .claude/agents/card-researcher.md && echo "  researcher: 축분장+앵커대조+압축반환 O"
grep -lE "동명이의|환각|PII|REMOVE|anchor_match=no" .claude/agents/card-validator.md && echo "  validator: 동명이의+환각+PII 검증 O"
grep -lE "card-research|트리거|명함" .claude/skills/card-research/SKILL.md && echo "  skill: 진입점 O"

echo "=== (3) 최종 md Contract 키 완비 (커맨드에) ==="
for k in harness subject_name subject_org subject_title subject_slug run_at run_id occupation_class axes_run total_facts anchor_dist axes_empty pii_excluded ; do
  grep -rq "$k" .claude/commands/card-research.md \
    && echo "  key $k O" || echo "  key $k MISSING ✗"
done

echo "=== (4) PII 윤리 경계가 3에이전트에 박혔는가 ==="
for f in card-parser card-researcher card-validator ; do
  grep -lqE "사생활|PII|수집 금지|공개 직업 정보|경계" ".claude/agents/$f.md" \
    && echo "  $f: PII 경계 명시 O" || echo "  $f: ✗ PII 경계 누락"
done

echo "=== (5) 도메인 무관·동명이의 핵심 개념 (의료고정 아님 / 앵커대조) ==="
grep -nE "의료.*고정|의학 전용|특정 도메인 고정" \
  .claude/agents/card-parser.md \
  | grep -vE "않는다|금지|아니|말라|마라|회피" \
  && echo "  ✗ 도메인 고정이 긍정 구현됨 — 계약 위반" \
  || echo "  도메인 무관 — OK"
grep -cE "anchor_match|동명이의|신원 앵커" .claude/agents/card-researcher.md .claude/agents/card-validator.md

echo "=== (6) 이식 이력 누출 부재 (결함 G — 0건이어야) ==="
grep -rnE "재구축|정밀 복제|review-watch|deep-search|사유원|토스|round-daily" \
  .claude/commands/card-research.md .claude/agents/card-*.md .claude/skills/card-research/SKILL.md \
  && echo "  ✗ 개발이력/타하네스 누출 — 결함 G" \
  || echo "  이력 누출 0건 — OK"
```

### 합격 기준

| 검사 | PASS 조건 |
|---|---|
| (1) 파일 존재 | 5개 전부 OK |
| (2) 책임 개념 | 5파일 각각 자기 책임 개념 실제 구현(grep은 힌트, 최종 판단은 너) |
| (3) Contract 키 | 13개 키 전부 O |
| (4) PII 경계 | parser·researcher·validator 3곳 전부 PII 경계 명시 O |
| (5) 도메인무관·앵커 | "도메인 무관 — OK" + researcher/validator 앵커대조 개념 ≥1 |
| (6) 이력 누출 | "이력 누출 0건 — OK" (결함 G. 특히 타 하네스명 0건) |

6개 검사 전부 PASS 면 자율 구축 성공. 사용자에게 검사 표 + **"각
파일을 명세 어디를 보고 어떻게 구현했는지 1줄 설명"** 을 덧붙여
보고하라(이해 기반 구현임을 보이기 위함). 그 후 종료한다.
(하네스를 실제 실행하지는 마라.)
