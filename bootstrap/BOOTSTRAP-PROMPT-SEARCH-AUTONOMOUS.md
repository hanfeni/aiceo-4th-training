# 작업 지시: `/deep-search` 검색 누락방지 하네스 자율 구축

너는 지금 이 하네스가 **설치되지 않은** 환경에 있다. 아래는 이 하네스가
**만족해야 하는 설계 명세**다. 원문 파일은 주어지지 않는다. **명세를
이해하고 네가 직접 파일을 작성해** 하네스를 구축하라.

받아쓰기가 아니다. 표현·문장·구조는 네가 정한다. 단 아래 **기능 계약·
불변식·결함 처방**은 한 줄도 빠뜨리지 말고 구현하라. 글자가 아니라
**계약을 지키는 것**이 합격선이다.

작업 중 사용자에게 질문하지 마라(명세가 완결돼 있다). `pwd`로 현재
프로젝트 루트를 확인하고 그 기준으로 경로를 정한다. 이 단계에서 하네스를
실제로 실행하지는 마라(구축까지만).

---

## 1. 목적 (WHY)

사용자가 **특정 검색 요청**(한 가지 주제·질문에 대한 웹 조사)을 하면,
**5개의 검색 에이전트가 동일한 범위를 똑같이 병렬 조사**하고,
오케스트레이터가 5개 결과를 **합집합으로 취합·정리한 마크다운 1개**를
산출한다.

**정체성 원칙 (반드시 구현):**

- **전수 중복 병렬 (분장 아님).** 5개 에이전트는 범위를 나눠 맡지
  않는다. **모두 같은 검색 범위를 똑같이** 독립 수행한다. 이것이 이
  하네스의 정체성이다.
- **왜 5중 중복인가**: LLM 의 WebSearch 는 같은 질의라도 실행마다
  결과가 다르다(비결정적 누락). 같은 범위를 5번 독립 수행해 **합집합**
  하면, 한 에이전트가 놓친 것을 다른 에이전트가 메운다. 누락 보완이
  목적이지 범위 분담이 아니다.
- **검색으로 끝나지 않는다 — 질문에 답한다.** 산출물은 링크 나열이
  아니라, **수집된 결과를 근거로 검색 주제(사용자 질문) 자체에 직접
  답하는 종합 서술**을 반드시 포함한다. "그래서 결론이 무엇인가"가
  빠진 결과 목록만으로는 이 하네스 실패다(§7 종합 답변 섹션 강제).
- **"정리"와 "우열 판정"은 다르다.** 비교·랭킹·주관적 우열("무엇이
  더 낫다/옳다")은 하지 않는다. 그러나 **수집된 사실을 종합해 질문에
  답하는 것**은 우열 판정이 아니라 이 하네스의 본분이다. 종합 답변은
  수집 결과에 근거가 있는 사실만으로 구성하고, 핵심 주장마다 어느
  결과(URL)에 근거했는지 출처를 단다(환각 방어와 동일 원리).

검색 요청이 비어 있으면 검색 주제를 한 번만 물어 받고, 그 외에는
질문 없이 끝까지 수행한다.

## 2. 만들 파일과 책임 (WHAT — 구현은 네 몫)

`.claude/` 하위에 커맨드 1 + 에이전트 1 + 스킬 1 = **3파일**. 경로:

| 파일 | 역할 (이 책임을 네 표현으로 구현) |
|---|---|
| `.claude/commands/deep-search.md` | 오케스트레이터. 공유 메타 산출 + 동일 input 5병렬 발사 + 합집합 취합 + md 1개 저장 |
| `.claude/agents/deep-search-agent.md` | 검색 일꾼. 배정된 **동일 범위**를 독립 수집, 압축 결과만 반환(파일 안 씀) |
| `.claude/skills/deep-search/SKILL.md` | 자연어 자동인식 진입점. 커맨드 실행순서로 연결 |

검색 에이전트는 5개를 **같은 정의 파일 1개**(`deep-search-agent`)로
호출한다(5개 별도 파일이 아니다 — 동일 작업이므로). 오케스트레이터가
이 1개 정의를 **동일 input 으로 5번 동시 호출**한다.

에이전트 정의는 Claude Code subagent 형식(YAML frontmatter `name`/
`description`/`tools`/`model` + 본문). `tools` 는 책임 수행에 필요한
최소로 네가 판단(검색 에이전트는 WebSearch/WebFetch면 충분, 파일은
안 쓴다 — §5 불변식 2).

## 3. 토폴로지 (순서·병렬성 고정, 서술은 네 몫)

```
Step 0  오케스트레이터: 공유 메타 산출 + 저장 경로 결정      [본체]
Step 1  deep-search-agent ×5   [동일 input · 단일 메시지 5병렬 · 전수 중복]
Step 2  오케스트레이터: 5결과 합집합 취합(URL dedup + agreement 집계)
Step 3  최종 md 1개 저장 + 작업 로그 + 보고
```

- **Step 0 공유 메타**: `now_kst`, `today`, `run_id`, `query`(정규화한
  검색 주제), `slug`(query 의 kebab-case 요약). 오케스트레이터가
  **한 번만** 산출해 5개 에이전트에 **완전히 동일하게** 주입한다.
  5개가 받는 input 은 글자 단위로 같아야 한다(그래야 "같은 범위를
  똑같이"가 성립). 어느 에이전트도 자체 `date`를 호출하지 않는다.
- **Step 1**: `deep-search-agent` 를 **단일 메시지 안에서 5개 동시**
  호출(병렬 계약. 직렬로 풀면 위반). 5개 input 은 **완전 동일**
  (범위 분담 표현을 어느 하나에도 넣지 마라 — 분장이 되어버린다).
- **Step 2**: 5개 에이전트의 반환(압축 결과 목록)을 오케스트레이터가
  **합집합**한다. dedup·agreement 규칙은 §4.
- **Step 3**: 취합 결과를 **마크다운 1개**로 저장. 그 외 마크다운
  파일은 만들지 않는다(에이전트는 파일을 쓰지 않는다 — §5 불변식 2).

## 4. 합집합 취합 규칙 (Step 2 — 이 계약 고정)

5개 에이전트 결과를 합칠 때:

- **URL 기준 dedup**: 같은 URL 은 1개 항목으로 합친다(정규화: 쿼리
  스트링·앵커·http/https·끝 슬래시 차이는 동일 취급).
- **agreement 수치**: 각 항목에 **몇 개 에이전트가 그 URL 을
  찾았는지**를 `agreement: N/5` 로 기록한다. 이것이 이 하네스의 핵심
  산출 특성이다:
  - `5/5` = 5개 에이전트 전원이 독립적으로 찾음 = 강한 신호(누락
    위험 낮음).
  - `1/5` = 한 에이전트만 찾음 = **누락 보완이 실제로 작동한 증거**
    (나머지 4개가 놓쳤으나 1개가 건짐). 버리지 말 것 — 이게 5중
    중복의 존재 이유다.
- 결과 정렬: `agreement` 내림차순(전원 합의 항목이 위), 동률이면
  관련도 순. **삭제·우열 판정 금지** — agreement 1/5 도 결과에 남긴다.
- **인용 무결성(환각 방어)**: URL·제목·한 줄 요지가 모두 있는 항목만
  취합한다. 셋 중 하나라도 없으면 그 항목 폐기(지어내지 말 것).
  5개 전부 결과가 0건이면 "0건"으로 정직하게 보고한다.
- **취합 후 종합**: dedup·agreement 집계로 끝내지 말고, 취합된
  결과를 근거로 검색 주제(질문)에 답하는 **종합 서술을 오케스트
  레이터가 작성**한다(§7-1 종합 답변). 형태는 질의에 맞춰 정하되
  근거 없는 문장은 넣지 않는다. 0건이면 "근거 없어 답 불가"로.

## 5. 불변식 (어기면 하네스 실패)

1. **공유 메타·input 완전 동일**: 5개 에이전트가 받는 input 은 글자
   단위로 같다. 어느 하나에도 "너는 X만", "1/5 범위" 같은 분담
   표현을 넣지 않는다(넣으면 전수 중복이 분장으로 무너진다).
2. **에이전트는 파일을 쓰지 않는다**: 검색 에이전트는 raw 파일을
   만들지 않는다(md 는 최종 1개만). 대신 오케스트레이터에 **압축
   결과 목록**만 반환한다: 항목당 `{url, title, 한 줄 요지(≤200자),
   근거 출처}`. **장문 본문·전체 페이지 덤프 금지**(컨텍스트 폭주
   방지 — 5개가 같은 범위라 중복이 많아 양이 커질 수 있다).
3. **단일 취합 지점**: 합집합·dedup·agreement 집계는 오케스트레이터
   1곳에서만 한다. 에이전트는 자기 결과만 반환하고 서로 합치지 않는다.
4. **결정적 단계 자가검증**: 최종 md 를 저장한 뒤, 저장 파일이 실제
   존재하고 agreement 표기가 모든 항목에 있는지 `grep` 등으로 **수행
   후 기계 확인**한 뒤에만 "저장 완료"로 보고한다(확인 없이 했다고
   가정·허위 기록 금지).

## 6. 결함 처방 (이미 비싸게 학습된 함정 — 회피 설계 필수)

아래는 유사 멀티에이전트 하네스를 여러 번 돌려 얻은 실패의 **일반
원리**다. 구체 사례·고유명은 의도적으로 뺐다(처방만 구현하라).

- **B. 컨텍스트 폭주 회피**: 다중 에이전트 결과가 오케스트레이터
  컨텍스트로 모이면 출력이 잘린다(예: 50K 절단). 이 하네스는 md 를
  1개만 만들어 에이전트가 파일을 안 쓰므로, **에이전트 반환을 압축
  결과 목록으로 강제**(불변식 2)해 폭주를 막는다. 항목 수가 많으면
  에이전트가 자기 단계에서 명백 중복·저관련을 1차 정리해 반환한다.
- **환각 방어**: URL·제목·요지 3종이 없는 항목 폐기(불변식 4의
  무결성). 에이전트가 검색 결과에 없는 URL·내용을 만들지 않는다.
  0건이면 0건으로 보고. **종합 답변 서술도 같은 원리** — 수집된
  결과에 근거 없는 문장을 넣지 않는다(근거 표시 강제, §7-1).
- **"수집하고 답을 안 한다" 회피**: 검색 하네스가 결과 목록만
  나열하고 사용자 질문 자체에는 답하지 않으면, 사용자는 링크 더미를
  직접 읽어야 한다 — 하네스가 일을 절반만 한 것이다. 산출물은
  **수집 결과를 근거로 질문에 직접 답하는 종합 서술을 맨 앞에
  필수**로 둔다(§7-1). 단 그 답의 형태를 도메인 고정 템플릿으로
  박지 말 것 — 질문 유형이 다양하므로 에이전트가 질의에 맞춰
  형태를 정한다. "정리만 하라"는 지시가 "답하지 마라"로 잘못
  읽히지 않도록 정의에 종합 답변을 명시 강제한다.
- **F. 비결정 단계 누락 회피**: 저장·집계 같은 결정적 단계는 정의에
  "하라"만 적지 말고 수행 후 grep 등으로 재확인하게 한다(불변식 4).
- **G. 이식 이력 누출 회피**: 네가 작성하는 3파일에 개발 이력·고유
  사례명·"재구축/복제" 같은 표현을 넣지 마라. 규칙의 **정당성
  근거**는 적되 "언제 누가 발견했는가"는 적지 않는다. 이 하네스는
  이식 가능한 산출물이어야 한다.
- **분장 오염 회피 (이 하네스 고유)**: 5개 input 동일성이 깨지면
  전수 중복이 분장으로 무너져 누락 보완 효과가 사라진다. 오케스트
  레이터가 5개를 호출할 때 input 을 변형·분할하지 않도록 명시한다.

## 7. 산출물 Contract (최종 md 1개 — 구조 고정, 표현은 네 몫)

저장 경로: `./specs/deep-search/<slug>/<today>/00_result.md`
(같은 날 재실행 시 `00_result_<run_id>.md` 누적 — 덮어쓰기 금지).
프로젝트 로컬, 전역 vault 금지.

최상단 YAML frontmatter 에 **다음 키를 전부** 포함(이름·구조 고정,
값은 실행마다 다름. Contract 외 키 추가 금지):

```
harness: deep-search
query: <정규화한 검색 주제>
slug: <slug>
run_at: <now_kst>
run_id: <run_id>
agents_run: 5
total_results: <dedup 후 고유 URL 수>
agreement_dist: { "5": N, "4": N, "3": N, "2": N, "1": N }
agents_empty: <0건 반환한 에이전트 수>
```

본문 필수 섹션(아래 5개 모두. **종합 답변이 맨 앞**, 나머지 순서 자유):

1. **종합 답변** *(필수·맨 앞·이 하네스의 핵심 산출)* — 수집된
   결과를 근거로 **검색 주제(사용자 질문) 자체에 직접 답하는 서술**.
   결과 목록을 읽지 않고 이 섹션만 봐도 "그래서 무엇인가"를 알 수
   있어야 한다.
   - **형태는 고정 템플릿이 아니다.** 질문은 매우 다양하게 들어온다
     (현황 파악, 사실 확인, 사건 경위, 옵션 조사, 방법 탐색 등).
     **질문 유형에 맞는 형태를 에이전트가 스스로 판단해** 작성하라
     — 현황형이면 핵심 사실 요약, 경위형이면 시간순 정리, 옵션형이면
     발견된 선택지 정리 등. 소제목·길이·구성은 주제에 맞춰 정한다.
     특정 도메인(주식·기술 등)에 박힌 고정 소제목을 쓰지 마라.
   - **근거 강제(환각 방어)**: 종합 답변의 모든 핵심 주장은 수집된
     결과에 근거가 있어야 한다. 주장마다 근거가 된 항목을 짧게 표시
     (예: 출처 매체/URL 또는 `[근거: 5/5·4/5 항목]`). 수집 결과에
     없는 사실을 만들어 넣지 마라. 결과가 0건이면 "수집된 근거가
     없어 답할 수 없음"이라고 정직하게 쓴다(추측 금지).
   - **우열 판정과의 경계**: "무엇이 더 낫다/옳다"는 주관 평가·랭킹은
     쓰지 않는다. 그러나 수집된 사실의 종합·요약·시간순 정리는 우열
     판정이 아니라 이 섹션의 본분이다. 전망·예측을 쓸 때는 하네스가
     판정하지 말고 "누가(어느 출처가) 그렇게 전망했다"는 인용 형태로.
2. **결과 목록** — agreement 내림차순 표/목록. 각 항목:
   `agreement N/5 | 제목 | URL | 한 줄 요지`. 1/5 항목도 포함.
3. **한눈 요약** — 검색 주제·고유 결과 수·5/5 합의 항목 수 1~2문장.
4. **누락 보완 효과** — agreement 1/5·2/5 항목 수와 그 의미(소수
   에이전트만 건진 = 단일 수행이었으면 놓쳤을 결과)를 1~2문장.
5. **수집 메모** — 0건 에이전트 수, 무결성 폐기 수, 한계(검색 도구
   제약 등). 데이터 없으면 0건으로 정직히.

결과 목록·정렬에는 우열·랭킹 판정을 쓰지 않는다(agreement 정렬만).
종합 답변 섹션은 위 1번 규칙(질의 적합·근거 강제·인용형 전망)을 따른다.

## 8. 수행 절차

1. `pwd` 로 루트 확인. `.claude/{commands,agents,skills/deep-search}`
   와 `.claude/history/daily` 디렉터리 생성.
2. §1~7 을 충족하는 **3개 파일을 네 표현으로 작성**해 각 경로에 Write.
   원문을 받지 않았으므로 베낄 것이 없다. 명세를 구현하라. 에이전트
   정의 말미에 "출력 직전 자가검증 체크리스트"를 네가 설계해 넣어라.
3. §9 자가검증 수행, 결과 표 보고.
4. 하네스를 실제로 실행하지 마라(구축까지만).

## 9. 자가검증 (기능 계약 충족도 — 글자가 아니라 계약)

1:1 비교는 하지 않는다(자율 작성이라 글자가 달라야 정상). 아래를
점검하고 PASS/FAIL 보고. FAIL 이면 그 파일을 계약에 맞게 고친 뒤
재점검한다.

```bash
echo "=== (1) 3파일 존재 ==="
for f in .claude/commands/deep-search.md \
  .claude/agents/deep-search-agent.md \
  .claude/skills/deep-search/SKILL.md ; do
  [ -f "$f" ] && echo "OK $f" || echo "FAIL $f MISSING"
done

echo "=== (2) 각 파일이 자기 책임 개념을 담는가 (개념 존재 — 글자 아님) ==="
grep -l "5.*병렬\|동일 input\|전수\|합집합\|agreement" .claude/commands/deep-search.md && echo "  cmd: 5병렬+동일input+합집합 개념 O"
grep -l "동일.*범위\|파일.*안 [쓰만]\|압축.*결과\|반환" .claude/agents/deep-search-agent.md && echo "  agent: 동일범위+파일미작성+압축반환 개념 O"
grep -l "deep-search\|트리거\|검색" .claude/skills/deep-search/SKILL.md && echo "  skill: 진입점 O"

echo "=== (3) 최종 md Contract 키 완비 (커맨드에) ==="
for k in harness query slug run_at run_id agents_run total_results agreement_dist agents_empty ; do
  grep -rq "$k" .claude/commands/deep-search.md \
    && echo "  key $k O" || echo "  key $k MISSING ✗"
done

echo "=== (4) 금지 개념 부재 (이 설계에 없어야 — 분장/우열/다중md) ==="
grep -rn "범위 분담\|분장\|너는 .*만 검색\|1/5 범위\|우열 판정\|랭킹\|raw 파일.*Write\|에이전트.*파일.*저장" \
  .claude/commands/deep-search.md .claude/agents/deep-search-agent.md \
  | grep -v "안 [함된]\|없다\|없음\|하지 않\|아니다\|금지\|않는다\|아님\|무너진\|분장이 되어\|분장으로" \
  && echo "  ✗ 금지 개념이 긍정적으로 구현됨 — 계약 위반" \
  || echo "  금지 개념 부재 — OK"

echo "=== (5) 이식 이력 누출 부재 (결함 G — 0건이어야) ==="
grep -rn "재구축\|정밀 복제\|review-watch\|토스\|사유원\|미래에셋\|4차\|6차" \
  .claude/commands/deep-search.md .claude/agents/deep-search-agent.md .claude/skills/deep-search/SKILL.md \
  && echo "  ✗ 개발이력/타하네스/고유사례 누출 — 결함 G" \
  || echo "  이력 누출 0건 — OK"

echo "=== (6) 종합 답변 섹션 강제 (질문에 답함 — 링크나열만 금지) ==="
grep -nE "종합 답변|질문.*답|주제.*답|결론" .claude/commands/deep-search.md \
  && echo "  종합 답변 개념 명세됨 (맨 앞·근거강제·질의적합인지 본문 직접 확인)" \
  || echo "  ✗ 종합 답변 섹션 미명세 — 링크 나열만 산출 (결함: 일을 절반만 함)"
grep -nE "고정 템플릿|도메인 고정|질의.*맞춰|질문 유형" .claude/commands/deep-search.md \
  && echo "  질의 적합(비고정 템플릿) 개념 명세됨" \
  || echo "  ✗ 종합 답변이 고정 템플릿일 위험 — 질의 적합 명세 누락"

### 합격 기준

| 검사 | PASS 조건 |
|---|---|
| (1) 파일 존재 | 3개 전부 OK |
| (2) 책임 개념 | 3파일 각각 자기 책임 개념을 실제 구현(grep은 힌트, 최종 판단은 너) |
| (3) Contract 키 | 9개 키 전부 O |
| (4) 금지 개념 | "금지 개념 부재 — OK" (분장·우열·에이전트파일저장이 긍정 구현 안 됨) |
| (5) 이력 누출 | "이력 누출 0건 — OK" (결함 G 회피. 특히 `review-watch`·`토스` 등 타 하네스/사례명 0건) |
| (6) 종합 답변 | 종합 답변 섹션이 산출물 정의에 **맨 앞 필수**로, **고정 템플릿 아닌 질의 적합**으로, **근거 강제**로 명세됨 (grep은 힌트 — 정의 본문을 직접 읽어 맨앞·근거강제·비고정 3요건 확인) |

6개 검사 전부 PASS 면 자율 구축 성공. 사용자에게 검사 표 + **"각
파일을 명세 어디를 보고 어떻게 구현했는지 1줄 설명"** 을 덧붙여
보고하라(이해 기반 구현임을 보이기 위함). 그 후 종료한다.
(하네스를 실제 실행하지는 마라.)
