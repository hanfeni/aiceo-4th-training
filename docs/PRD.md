## 1. 검색 문서 Feasibility PoC

### 1.1 개요

서울대 AI CEO 4기 2회차 강의(2026-05-22)의 데이터 연결 실습에서, 수강생이 자기 직군에 맞는 5개 도메인 중 1개를 택해 **검색3종(BM25 / 임베딩 / 하이브리드)** 을 실습한다. 본 기능은 그 검색 실습이 **실제 코퍼스로 교육적으로 성립하는지(feasibility)** 를 강의 전(D-4)에 정성적으로 검증하는 PoC를 정의한다.

라이선스·무인증 접근·본문 글자수는 vault 노트의 6차 5-에이전트 검증에서 실측 완료되었다(상세: `~/Library/Mobile Documents/iCloud~md~obsidian/Documents/Themes/lectures/2026-05-aiceo-data-connection-samples.md` §6차). 본 PoC가 검증하는 미검증 항목은 단 하나다: **실제로 인덱싱해 검색3종을 돌렸을 때 세 방식의 결과가 의도대로 갈리는가**.

### 1.2 배경

- 6차 검증에서 5개 도메인 각각이 "구조화 데이터(SQL/그래프) + 짝 서술 문서(검색3종)"로 완결됨이 확정되었다. 본 PoC는 이 중 **검색 트랙(짝 서술 문서)** 의 feasibility만 대상으로 한다.
- 강의 핵심 메시지 중 하나는 "BM25 우세 / 임베딩 우세 / 하이브리드 우세 질의가 극명히 갈린다"이다(2회차 노트 문서 트랙 질의 세트 참조). 이 메시지가 **실코퍼스에서 재현되지 않으면 강의 데모가 무너진다**. 라이선스 페이지만으로는 안 보였던 함정을 실 curl이 잡아냈듯, 인덱싱·검색 실행으로만 잡히는 함정이 남아 있을 수 있다.
- 강의는 30명이 강의장에서 재현하므로, PoC는 무인증·사전 번들 전제와 한국어 처리 제약을 그대로 반영해야 한다.

### 1.3 스코프 정직성 (필독 — 결과물에 명시 강제)

본 PoC는 **정답지(ground truth) 없이 진행**한다(사용자 결정). 따라서:

- 본 PoC는 **정량 검색 품질 평가가 아니다.** nDCG·MRR·Recall@k 등의 지표를 산출하지 않는다.
- 본 PoC의 산출 판정은 **사람이 top-k 결과를 보고 내리는 정성 feasibility 판정**이다("이 도메인은 강의 데모로 적합/조건부/부적합").
- 모든 리포트 산출물 표지/헤더에 **"본 결과는 정성 feasibility 판정이며 정량 검색 품질 평가가 아님"** 문구를 명시한다(인수 기준 AC-7).
- 이 한계는 의도된 것이며, 정량 평가는 §1.7 비목표에 명시한다.
- OpenSearch 하이브리드 부스팅이 LLM 메타 품질에 종속되므로, 메타 라벨링이 적용된 도메인의 정성 판정 시 "하이브리드 우세가 메타 부스팅에 기인한 것인지" 단서를 함께 명시한다(메타 품질 종속 정성 리스크).

### 1.4 사용자 스토리

- US-1: **강사(김두환)** 로서, 강의 전에 5개 도메인 각각의 검색3종이 실코퍼스에서 의도대로 갈리는지 확인하고 싶다. 그래야 강의 당일 데모가 무너지지 않는다.
- US-2: **강사** 로서, 도메인별 코퍼스가 "문서 몇 개 / 평균 몇 글자·토큰·형태소 / 총 규모"인지 정확한 숫자로 알고 싶다. 그래야 청킹·임베딩 비용·실습 시간을 산정할 수 있다.
- US-3: **강사** 로서, "강의 데모로 적합/조건부/부적합" 도메인 판정을 정성 근거와 함께 받고 싶다. 그래야 부적합 도메인을 강의 전에 교체하거나 질의를 보정할 수 있다.
- US-4: **수강생(30명)** 으로서, 내가 택한 도메인에서 세 검색 방식의 차이를 강의장 노트북에서 무인증으로 재현하고 싶다.

### 1.5 기능 요구사항 (FR)

#### 코퍼스 수집

- **FR-1.1**: 5개 도메인 각각 실코퍼스를 수집한다. 목표 건수는 도메인당 300~500건.
  1. 상권/소상공인: 상가(상권)정보 + 정책뉴스(`policyNewsView`, `article_body` 글자수 ≥ 800 필터) 소상공인 서브셋
  2. 의료/제약: 약가마스터 + 의약품안전나라 허가상세(전문약, `품목기준코드` = 약가마스터 JOIN 키)
  3. 금융/연금/고용: 국민연금사업장 + 정책뉴스 고용·금융 서브셋
  4. 법률: legalize-kr 법령·판례(frontmatter SQL + 참조조문 그래프 + commit 시계열, 단독 완결 — 짝 문서 불필요)
  5. 정책/거버넌스: data.go.kr 무인증 세출예산 CSV + 정책브리핑 보도자료 본문
- **FR-1.2**: 봇 차단 소스(의약품 허가상세, 정책뉴스)는 300~500건 목표를 강제하지 않고 **"최소 80건" 가변 목표**를 적용한다. 80건 미만이면 해당 도메인 판정에 "코퍼스 부족" 플래그를 명시한다.
- **FR-1.3**: 수집 데이터는 `policyNewsView` 경로를 사용한다. `pressReleaseView`(보도자료)는 본문이 HWP-iframe에 갇혀 ~40자만 추출되므로 검색 코퍼스 수집에 사용하지 않는다(6차 검증 함정 #2).
- **FR-1.4**: 본문 사진 캡션(`(ⓒ뉴스1...)` 등 제3자 저작권 표기) 및 HTML 태그는 정규식으로 제거한다(라이선스·노이즈 제거).

#### 코퍼스 3축 규모 스펙 (사용자 핵심 요청)

- **FR-2.1**: 도메인별로 다음 3축 규모 스펙을 산출한다.
  - 문서 1개 평균 **글자수** (한국어 음절 기준)
  - 문서 1개 평균 **추정 토큰수** — 임베딩에 실제 사용할 모델의 토크나이저로 실측(추정치가 아닌 토크나이저 실행 결과)
  - 문서 1개 평균 **한국어 형태소 수** — `kiwi` 형태소 분석기 실측
- **FR-2.2**: 도메인별 **총 규모**를 `평균값 × 전체 문서 개수`로 산출해 표로 제시한다(글자/토큰/형태소 3축 각각의 총합).
- **FR-2.3**: 3축 스펙 산출 시 사용한 토크나이저 모델명·버전, `kiwi` 버전을 결과에 명시한다(재현성).

#### 청킹

- **FR-3.1**: 청킹 정책을 강제한다 — **1000 토큰 슬라이딩 윈도우, overlap 100 토큰**.
- **FR-3.2**: 단일 문서를 통째로 1개 임베딩 벡터로 만드는 것을 금지한다(예: 의약품 허가상세 전문약 27,000자+ 단일 벡터 금지). 청크 단위로 인덱싱한다.
- **FR-3.3**: 청킹 결과 도메인별 총 청크 개수를 3축 스펙 표에 함께 보고한다.

#### 인덱싱

- **FR-4.1**: BM25 인덱스를 **OpenSearch 인덱스(BM25 유사도)** 로 구축한다. 한국어 토크나이징은 OpenSearch 네이티브 **Nori analyzer(`decompound_mode=mixed`)** + **ngram(1-2gram) analyzer**를 병용한다(복합어 분해 + 부분일치 커버). `kiwi`는 인덱싱 토크나이저로 사용하지 않으며 FR-2.1 형태소 *측정* 용도로만 잔존한다(인덱싱과 측정 역할 분리). OpenSearch 인덱스 매핑·analyzer 설정은 `poc/opensearch/mapping.json`으로 버전관리한다(참조: `requirement/커뮤니티-AI검색/04-opensearch-index.md` §7).
- **FR-4.2**: 임베딩 인덱스를 OpenSearch `knn_vector`(HNSW) 필드로 구축한다. 임베딩 백엔드는 **OpenAI embedding(`text-embedding-3`)** 을 사용한다. Claude(Anthropic) 임베딩은 API 부재로 후보에서 삭제한다. knn 벡터 **차원은 S0에서 비가역적으로 사전 확정**되며(인덱스 매핑에 dimension 고정), 이후 중간 전환은 인덱스 재생성 없이는 불가하다.
- **FR-4.3**: OpenAI 임베딩 키가 전무한 경우에 한해 로컬 `bge-m3`(1024차원)로 진행한다. 단, **S0에서 처음부터 1024차원으로 인덱스를 생성하는 사전 결정 경로**로만 허용하며, 런타임 중간 fallback 전환(차원 변경)은 금지한다(knn 차원 락인). bge-m3는 S0 사전 결정 경로 외에는 사용하지 않는다.
- **FR-4.4**: 슬라이스 S0에서 **비가역 결정 게이트**를 둔다 — (1) 임베딩 백엔드(OpenAI `text-embedding-3` vs 사전결정 로컬 `bge-m3`), (2) knn 벡터 차원, (3) LLM 메타 라벨링 백엔드 가용성 3종을 인덱싱 착수 전에 점검·결정·기록한다. 결정 후에는 인덱스 재생성 없이 변경 불가하다. 결정된 차원은 OpenSearch 매핑·임베딩 호출·검색 쿼리 전 구간에 단일 출처로 전파한다.
- **FR-4.5**: 정제 직후·청킹 직전에 **문서 단위 LLM 메타 라벨링** 단계를 둔다. 문서별로 `main_category` / `mid_category` / `sub_category` / `description` / `keywords` / `system_alert`를 LLM으로 분류해 OpenSearch 문서 메타 필드로 인덱싱한다(참조: `requirement/커뮤니티-AI검색/05-llm-metadata.md` 1단계 분류 패턴). 이 메타 필드는 검색3종 중 하이브리드 부스팅에 사용한다(`keywords^3`·`description^3`, 참조: 04-opensearch-index.md §4).
- **FR-4.6**: LLM 메타 라벨링 백엔드 키 가용성은 FR-4.4 S0 게이트가 임베딩 키와 함께 결정한다. 메타 키가 부재할 경우 (a) 메타 라벨링을 skip 하고 리포트에 "메타 부스팅 미적용" 단서를 명시하거나, (b) 작업을 중단하고 HITL로 사용자 결정을 받는다. 어느 경로든 S0 결정 기록에 남긴다.
- **FR-4.7**: 메타 라벨링은 **일회성 사전 단계로 발굴한 `meta_schema.json`** 을 런타임에서 읽기 전용으로 로드해 수행한다. 메타 스키마 발굴: 5개 도메인 각각 10~100 샘플 N세트를 LLM에 돌려 메타 후보(분류 체계·필드)를 제안받고, 취합·수렴해 `meta_schema.json`을 선정한다(1회차 5-에이전트 수렴의 메타 버전). 발굴 도구·발굴 코퍼스는 **런타임 파이프라인 밖**의 일회성 산출물이며, 런타임은 산출된 스키마만 읽는다.
- **FR-4.8**: 검색 엔진은 PoC가 소비하는 **로컬 OpenSearch 컨테이너(Docker, `localhost:9200`)** 이다. PoC는 이 stateful 로컬 서버를 소비만 하며, 컨테이너 라이프사이클(기동·헬스체크·종료)은 `run_poc.sh`가 관리한다. OpenSearch 연결·인덱싱 코드는 라이프사이클 관리와 경계를 분리하고, indexer에 연결을 주입한다(AI5).

#### 검색3종 실행

- **FR-5.1**: 세 가지 검색을 실행한다 — (a) BM25, (b) 임베딩, (c) 하이브리드.
- **FR-5.2**: 하이브리드는 **OpenSearch 네이티브 하이브리드**로 결합한다 — `multi_match`(BM25, Nori/ngram 필드 대상) + `knn` boost를 단일 `bool.should` 쿼리로 결합한다(참조: `requirement/커뮤니티-AI검색/04-opensearch-index.md` §3·§4). knn boost·필드 가중치(`TITLE^4`·`description^3`·`keywords^3`·`CONTENT^2` 등)·`tie_breaker`를 grid로 튜닝하고, 채택한 설정값을 결과에 기록한다. RRF 자체 구현은 비목표(§1.7)로 이동한다.
- **FR-5.3**: 질의 세트는 도메인별로 구성하되 **절반은 6차 검증 설계형 질의**(BM25/임베딩/하이브리드 우세가 의도된 질의), **절반은 수강생이 던질 법한 거친 자연어형 질의**로 구성한다.
  - 설계형 예시(정책/거버넌스 등): BM25 우세 `"중대재해처벌법 시행령"`, 임베딩 우세 `"기업이 친환경 전환할 때 받는 정부 지원"`, 하이브리드 우세 `"2026년 소상공인 R&D 세액공제 변경"`.
- **FR-5.4**: 도메인 × 질의에 대해 세 방식 각각의 **top-5 결과 매트릭스**를 산출한다(질의별로 세 방식의 상위 5건을 나란히 비교 가능한 형태).

#### Feasibility 리포트

- **FR-6.1**: 도메인별로 **정성 feasibility 판정**을 산출한다 — `강의 데모 적합` / `조건부` / `부적합` 3단계. 각 판정에 정성 근거(어느 질의에서 세 방식이 갈렸는지/안 갈렸는지)를 첨부한다.
- **FR-6.2**: `조건부` 판정에는 강의 전 보정 액션(질의 교체, 코퍼스 보강, 필터 조정 등)을 구체적으로 명시한다.
- **FR-6.3**: 리포트 표지·각 도메인 섹션 헤더에 §1.3의 정성 한계 문구를 명시한다.

### 1.6 비기능 요구사항 (NFR)

- **NFR-1 (시간 제약)**: 강의일이 2026-05-22이고 작성 기준일이 2026-05-18이므로 D-4. PoC는 이 기간 내 완료 가능한 범위로 설계한다(전체 코퍼스가 아닌 도메인당 300~500건/최소 80건 표본).
- **NFR-2 (재현성)**: 30명이 강의장 노트북에서 재현하므로 무인증 접근을 1순위로 한다. 사전 번들이 필요한 소스(의약품 허가상세, 정책뉴스, legalize-kr 스냅샷)는 PoC 단계에서 번들 수집 방식을 검증한다.
- **NFR-3 (한국어 처리)**: 한국어 처리 역할을 분리한다 — **인덱싱 토크나이징은 OpenSearch Nori analyzer(`decompound_mode=mixed`) + ngram(1-2gram)** 가, **형태소 *측정* 은 `kiwi`** 가 담당한다(FR-4.1·FR-2.1). 청킹 토큰 계산은 임베딩 모델 토크나이저로 한국어를 정확히 처리한다.
- **NFR-4 (API 비용 최소화)**: API 호출 비용을 2축으로 최소화한다 — (1) **임베딩**: 청크 단위 배치 호출, 청크 `text_hash` 기준 중복 캐싱, 표본 규모 상한 준수. (2) **LLM 메타 라벨링**(신규 비용 축): **문서 단위 캐싱**으로 동일 문서 재호출을 방지한다. 메타 캐시 키 = `doc_meta_hash = sha256(정제후 문서 본문)` 으로, 청크 `text_hash`와 **별도 입자**(문서 단위 vs 청크 단위)다. OpenAI 키 전무 시 임베딩은 비용 0 로컬 `bge-m3` 사전결정 경로(FR-4.3), 메타 키 부재 시 메타 skip(FR-4.6).
- **NFR-5 (배포 안전)**: PoC는 공개 배포 리포의 `poc/` 하위에 둔다. **버전관리 스키마 산출물**: OpenSearch 인덱스 매핑(`poc/opensearch/mapping.json`)과 메타 스키마(`meta_schema.json`)는 커밋한다. **`.gitignore` 제외 대상**: 수집 코퍼스 데이터 파일, API 키(`.env` 등), OpenSearch 데이터 볼륨, 메타 라벨링 발굴 코퍼스, 메타 라벨링 API 키. 리포에는 코드·버전관리 스키마·정성 리포트만 커밋한다.
- **NFR-6 (스냅샷 안정성)**: legalize-kr는 force-push 공지가 있으므로 PoC 수집 시점의 커밋 해시를 고정·기록한다(재현성).

### 1.7 비목표 (의도적 제외)

- **정량 검색 품질 평가** (nDCG / MRR / Recall@k 등): 정답지가 없어 산출 불가하며, 본 PoC는 정성 feasibility 판정만 수행한다(§1.3).
- **프로덕션 검색 시스템**: 본 PoC는 강의 feasibility 판정 + 교보재 목적이며, 운영 검색 서비스가 아니다.
- **SQL / 그래프 트랙 실습 코드**: 6차 검증의 구조화 데이터 트랙(SQLite 스키마, 그래프 변환)은 별도 작업이다. 본 PoC는 검색 트랙 feasibility만 다룬다.
- **5개 도메인 전체 코퍼스 수집**: 표본(300~500건/최소 80건)만 다룬다. 전체 수백만 행 수집은 범위 밖.
- **RRF(Reciprocal Rank Fusion) 자체 구현**: 하이브리드 결합은 OpenSearch 네이티브 하이브리드(`multi_match` + `knn` boost 단일 쿼리)로 대체한다(FR-5.2). RRF 융합 로직을 PoC가 직접 구현하지 않는다.

### 1.8 인수 기준 (AC)

- **AC-1**: 5개 도메인 각각에 대해 실코퍼스가 수집되어 있고, 도메인별 수집 건수가 기록된다. 봇 차단 소스(의약품 허가상세, 정책뉴스)는 최소 80건 이상이거나 "코퍼스 부족" 플래그가 명시된다.
- **AC-2**: 도메인별 3축 규모 스펙 표가 산출된다 — 평균 글자수 / 평균 토큰수(토크나이저 실측) / 평균 형태소수(kiwi 실측) / 전체 문서 개수 / 3축 총 규모 / 총 청크 개수. 토크나이저·kiwi 버전이 명시된다.
- **AC-3**: 청킹이 1000토큰·overlap 100으로 적용되었고, 단일 문서 통짜 임베딩이 없음을 확인할 수 있다.
- **AC-4**: OpenSearch 인덱스(BM25 + Nori `decompound_mode=mixed` + ngram 1-2gram) + knn 임베딩 필드가 구축되고, S0에서 (1) 임베딩 백엔드, (2) knn 벡터 차원, (3) LLM 메타 라벨링 백엔드 3종의 비가역 결정이 기록된다. 결정된 차원이 OpenSearch 매핑·임베딩 호출·검색 쿼리에 단일 출처로 전파됨을 확인할 수 있다.
- **AC-4b**: OpenSearch 인덱스 매핑(`poc/opensearch/mapping.json`)과 메타 스키마(`meta_schema.json`)가 버전관리 산출물로 커밋되어 있다. OpenSearch 데이터 볼륨·메타 발굴 코퍼스·메타 키가 `.gitignore`로 제외되어 커밋되지 않음을 검증할 수 있다.
- **AC-4c**: 정제 직후·청킹 직전 문서 단위 LLM 메타 라벨링(`main_category`/`mid_category`/`sub_category`/`description`/`keywords`/`system_alert`)이 수행되어 OpenSearch 메타 필드로 인덱싱되거나, 메타 키 부재 시 skip + 리포트 단서 명시 또는 HITL 중단이 S0 기록에 남는다. 메타 캐싱이 `doc_meta_hash`(문서 단위, 청크 `text_hash`와 별도 입자)로 적용됨을 확인할 수 있다.
- **AC-5**: 도메인 × 질의 top-5 결과 매트릭스가 산출되며, 질의 세트가 6차 설계형 절반 + 자연어형 절반으로 구성됨을 확인할 수 있다. 하이브리드는 OpenSearch 네이티브(`multi_match` + `knn` boost)로 결합되며, 채택한 knn boost·필드 가중치·`tie_breaker` grid 설정이 기록된다.
- **AC-6**: 도메인별 `적합 / 조건부 / 부적합` 정성 판정이 정성 근거와 함께 산출된다. `조건부`에는 보정 액션이 명시된다.
- **AC-7**: 모든 리포트 산출물에 "본 결과는 정성 feasibility 판정이며 정량 검색 품질 평가가 아님" 문구가 표기된다.
- **AC-8**: 수집 코퍼스 데이터·API 키가 `.gitignore`로 제외되어 커밋되지 않는다. PoC 코드와 정성 리포트만 `poc/` 하위에 커밋된다.

### 1.9 영향 범위

#### 영향받는 엔드포인트 (API 라우트)

- 본 PoC는 외부 API를 **소비**만 한다(신규/수정 서버 엔드포인트 없음 — 이 리포는 강의 배포 리포로 자체 API 서버가 없음). 다만 **로컬 OpenSearch 컨테이너(`localhost:9200`)에 의존**한다 — PoC가 소비하는 stateful 로컬 서버이며, 컨테이너 라이프사이클(기동·헬스체크·종료)은 `run_poc.sh`가 관리한다(신규 외부 서버 엔드포인트는 아님). 소비 대상:
  - 검색 엔진: 로컬 OpenSearch Docker 컨테이너(`localhost:9200`) — 인덱싱·검색3종·하이브리드 쿼리 소비
  - 임베딩: OpenAI Embeddings API(`text-embedding-3`) (키 전무 시 S0 사전결정 로컬 `bge-m3` 1024d, 외부 호출 없음 — 중간 전환 불가)
  - LLM 메타 라벨링: 문서 단위 분류 LLM API (메타 키 부재 시 skip 또는 HITL — FR-4.6)
  - 코퍼스 수집: data.go.kr 무인증 fileData(상가/약가/세출예산), korea.kr `policyNewsView`/정책브리핑, 의약품안전나라 허가상세, GitHub `legalize-kr` git clone

#### 스키마 변경

- 데이터베이스(RDB) 스키마 변경 없음. 단, 1·2차 PASS의 "스키마 없음" 전제가 일부 깨져 **버전관리 스키마 산출물**이 신규로 정의된다:
  - `poc/opensearch/mapping.json` — OpenSearch 인덱스 매핑(필드 타입, Nori/ngram analyzer 설정, `knn_vector` 차원·HNSW 파라미터). 버전관리 커밋 대상.
  - `meta_schema.json` — LLM 메타 라벨링 분류 체계·필드 정의(일회성 발굴 산출물). 런타임은 읽기 전용 로드. 버전관리 커밋 대상.
- `.gitignore` 제외 대상(로컬 산출물·데이터):
  - 수집 코퍼스 원본 (도메인별 디렉터리)
  - 청크 메타데이터 (문서ID·청크ID·오프셋·청크 `text_hash`)
  - 문서 단위 메타 캐시 (`doc_meta_hash` 키, 청크 `text_hash`와 별도 입자)
  - OpenSearch 데이터 볼륨 (컨테이너 인덱스 데이터)
  - 메타 라벨링 발굴 코퍼스 (일회성 발굴 입력)
  - API 키 (임베딩·메타 라벨링 `.env` 등)

#### UI 변경

- UI 페이지·컴포넌트 변경 없음. 산출물은 CLI 실행 결과 + 마크다운 정성 리포트(3축 스펙 표, top-5 결과 매트릭스, 도메인 판정표)이다.

### 1.11 신규 STRUCTURAL 작업 (AI5~AI8)

인덱스 스택 재정의로 신규 STRUCTURAL 작업 4종이 추가된다(architect 조건부 PASS 권고). PoC 정체성·AI1~AI4는 보존된다.

- **AI5 (OpenSearch 라이프사이클·연결 경계 분리)**: 컨테이너 라이프사이클(기동·헬스체크·종료, `run_poc.sh`)과 OpenSearch 연결을 경계 분리하고, indexer에 연결을 주입한다(FR-4.8).
- **AI6 (S0 3종 비가역 결정·차원 전파)**: S0 게이트에서 임베딩 백엔드·knn 차원·메타 라벨링 백엔드 3종을 비가역 결정·기록하고, 결정된 차원을 OpenSearch 매핑·임베딩 호출·검색 쿼리 전 구간에 단일 출처로 전파한다(FR-4.4).
- **AI7 (LLM 메타 라벨링)**: 정제 직후·청킹 직전 **문서 단위**로 메타 라벨링을 수행하고, `doc_meta_hash`(청크 `text_hash`와 별도 입자)로 캐싱한다(FR-4.5·FR-4.6·NFR-4).
- **AI8 (스키마 JSON 버전관리·발굴 분리)**: `poc/opensearch/mapping.json`·`meta_schema.json`을 버전관리 산출물로 커밋하고, 메타 스키마 발굴을 런타임 파이프라인 밖 일회성 단계로 분리하며, OpenSearch 데이터 볼륨·발굴 코퍼스·메타 키를 `.gitignore`에 추가한다(FR-4.7·NFR-5).
