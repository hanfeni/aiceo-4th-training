# Use Cases: 검색 문서 Feasibility PoC

> Based on [PRD](../PRD.md) §1 검색 문서 Feasibility PoC (v1.0, 2026-05-18)

---

## 문서 성격 (필독)

본 PoC는 **웹 서비스가 아니라 CLI 검증 파이프라인**이다. 따라서 본 use-case 문서의 시나리오는 사용자 클릭 플로우가 아니라 **파이프라인 단계별 실행·실패·복구 시나리오**다.

- **주 Actor**: 강사(PoC 실행자, 김두환). CLI 파이프라인을 실행하는 사람.
- **산출물 소비자**: 강의 설계자(= 강사 본인의 후속 역할). 정성 feasibility 리포트를 보고 강의 데모 도메인을 확정·교체·보정한다.
- **재현 Actor (간접)**: 수강생 30명. PoC 자체는 실행하지 않으나, PoC가 검증하는 무인증·사전 번들 전제의 수혜자이며 재현성 제약의 근거다.

### 파이프라인 단계 (UC 매핑 기준)

```
S0  임베딩 키 게이트 (백엔드 결정·기록)
 │
 ├─ 수집(collect)        → 5개 도메인 실코퍼스
 ├─ 3축 스펙(spec)       → 글자/토큰/형태소 × 문서개수 + 총 규모 + 청크수
 ├─ 청킹(chunk)          → 1000토큰 / overlap 100
 ├─ 인덱싱(index)        → BM25(kiwi) + 임베딩(API 또는 bge-m3)
 ├─ 검색3종(search)      → BM25 / 임베딩 / 하이브리드 RRF
 └─ 리포트(report)       → 도메인별 정성 feasibility 판정
```

### 스코프 정직성 (모든 UC에 적용되는 불변 제약)

- 본 PoC는 **정답지(ground truth) 없이 진행**한다. 정량 지표(nDCG/MRR/Recall@k)를 산출하지 않는다.
- 모든 판정은 **사람이 top-k 결과를 보고 내리는 정성 feasibility 판정**이다.
- 모든 리포트 산출물에 "본 결과는 정성 feasibility 판정이며 정량 검색 품질 평가가 아님" 문구 표기 (AC-7) — 이 제약은 UC-9에서 별도 검증된다.

### 5개 도메인 (UC 전반에서 참조)

| # | 도메인 | 코퍼스 소스 | 봇차단 여부 |
|---|--------|-------------|-------------|
| D1 | 상권/소상공인 | 상가(상권)정보 + 정책뉴스 소상공인 서브셋 | 정책뉴스 봇차단 |
| D2 | 의료/제약 | 약가마스터 + 의약품안전나라 허가상세(전문약) | 허가상세 봇차단 |
| D3 | 금융/연금/고용 | 국민연금사업장 + 정책뉴스 고용·금융 서브셋 | 정책뉴스 봇차단 |
| D4 | 법률 | legalize-kr 법령·판례(단독 완결, 짝 문서 불필요) | git clone, force-push 리스크 |
| D5 | 정책/거버넌스 | data.go.kr 무인증 세출예산 CSV + 정책브리핑 보도자료 | 무인증 우선 |

## UC-1: 임베딩 키 게이트 통과 후 전체 파이프라인 정상 완주

**Actor**: 강사(PoC 실행자)
**Preconditions**:
- 리포 `poc/` 하위에 PoC 파이프라인 코드가 존재한다.
- OpenAI 또는 Claude(Anthropic) 임베딩 API 키가 `.env` 등에 설정되어 사용 가능하다.
- `kiwi` 형태소 분석기, BM25 구현, 임베딩 클라이언트 의존성이 설치되어 있다.
- 5개 도메인 외부 데이터 소스(data.go.kr, korea.kr policyNewsView, 의약품안전나라, GitHub legalize-kr)가 접근 가능하다.

**Trigger**: 강사가 D-4 시점(2026-05-18)에 PoC 파이프라인 진입 명령(S0 게이트)을 실행한다.

### Primary Flow (Happy Path)
1. 강사가 S0 임베딩 키 게이트를 실행한다.
2. 시스템이 API 키 가용성을 점검한다 — OpenAI 또는 Claude 키가 유효함을 확인한다.
3. 시스템이 사용할 임베딩 백엔드를 "API(모델명·버전 명시)"로 결정하고 게이트 결과에 기록한다(AC-4).
4. 강사가 수집 단계를 실행한다 — 5개 도메인 각각에 대해 표본 스파이크(소량, 예: 도메인당 10건)로 수집 경로 유효성을 먼저 확인한다.
5. 스파이크가 5개 도메인 모두 성공하면 본 수집을 실행해 도메인당 300~500건(봇차단 소스는 최소 80건)을 수집한다.
6. 시스템이 본문 사진 캡션(`(ⓒ뉴스1...)`)·HTML 태그를 정규식으로 제거한다(FR-1.4).
7. 시스템이 도메인별 3축 규모 스펙(글자/토큰/형태소 평균 × 문서개수 + 총 규모)을 산출한다(UC-2 위임).
8. 시스템이 1000토큰·overlap 100 슬라이딩 윈도우로 청킹하고, 통짜 단일 벡터 생성이 없음을 보장한다(FR-3.1/3.2).
9. 시스템이 BM25(kiwi 토크나이저) 인덱스와 임베딩 인덱스를 구축한다(FR-4.1/4.2).
10. 시스템이 도메인 × 질의에 대해 검색3종(BM25/임베딩/하이브리드 RRF)을 실행해 top-5 매트릭스를 산출한다(UC-3 위임).
11. 시스템이 도메인별 정성 feasibility 판정 리포트를 산출한다(UC-4 위임).
12. 모든 리포트 산출물 표지·헤더에 정성 한계 문구가 표기된다(AC-7).

**Postconditions**:
- 5개 도메인 전부에 대해 수집→스펙→청킹→인덱싱→검색3종→판정이 완주된 상태.
- `poc/` 하위에 PoC 코드와 정성 리포트(마크다운)만 커밋 가능 상태. 코퍼스 원본·인덱스·`.env`는 `.gitignore`로 제외(AC-8).
- S0 게이트 결정 기록, 3축 스펙 표, top-5 매트릭스, 도메인 판정표가 존재한다.

### Alternative Flows
- **UC-1-A1: 임베딩 API 키 없음 → 로컬 bge-m3 fallback** — S0 게이트에서 API 키 미가용 시.
  1. S0 게이트 실행 (Primary 1번과 동일).
  2. 시스템이 OpenAI·Claude 키 모두 미가용임을 확인한다.
  3. 시스템이 로컬 `bge-m3` 모델로 자동 fallback 결정하고, 게이트 결과에 "백엔드=로컬 bge-m3, 외부 호출 없음, 비용 0"으로 기록한다(FR-4.3/4.4).
  4. 로컬 `bge-m3` 모델이 정상 로드/다운로드된다.
  5. Primary 4번부터 동일하게 진행하되 임베딩 인덱스 구축에 `bge-m3`를 사용한다.
  6. (이후 Primary와 동일하게 완주)
- **UC-1-A2: 일부 도메인만 완주, 나머지 코퍼스 부족 플래그** — 봇차단 소스 80건 미달 도메인이 일부 존재.
  1. Primary 1~6 진행.
  2. D2 또는 D1/D3의 봇차단 서브셋이 80건 미만으로 수집됨.
  3. 시스템이 해당 도메인 판정에 "코퍼스 부족" 플래그를 명시한다(FR-1.2/AC-1).
  4. 코퍼스 충족 도메인은 정상 판정, 부족 도메인은 "코퍼스 부족" 플래그 + 가능한 범위 정성 관찰로 리포트에 포함된다.

### Error Flows
- **UC-1-E1: 스파이크 단계에서 특정 도메인 수집 경로 전면 실패** — 스파이크 10건도 0건이 나오는 경우.
  1. Primary 4 진행 중 한 도메인의 스파이크가 0건.
  2. 시스템이 해당 도메인 수집 실패를 기록하고(원인: HTTP 차단/스키마 변경/소스 폐기 — UC-6 위임), 게이트로 강사에게 보고한다.
  3. 강사가 해당 도메인을 제외하고 나머지 4개로 파이프라인을 진행할지 결정한다(HITL).
  4. 결정된 범위로 Primary 5번 이후 진행. 제외 도메인은 리포트에 "수집 실패 — feasibility 판정 불가" 사유와 함께 기록된다.
- **UC-1-E2: S0 게이트 자체 실행 불가** — 의존성 미설치/환경 깨짐.
  1. 강사가 S0 게이트 실행.
  2. `kiwi`/임베딩 클라이언트/BM25 의존성 import 실패.
  3. 시스템이 누락 의존성을 명시한 에러로 즉시 중단(파이프라인 진입 차단).
  4. 강사가 의존성 설치 후 재실행. (게이트는 멱등 — 재실행 안전해야 함)

### Edge Cases
- **UC-1-X1**: 5개 도메인 중 일부는 API 백엔드, 일부는 bge-m3로 혼용 불가 — S0에서 백엔드는 파이프라인 전역 단일 결정이어야 하며, 도메인별 상이한 임베딩 백엔드 사용 시 검색3종 비교 무효(리포트에 백엔드 단일성 기록 강제).
- **UC-1-X2**: 게이트 통과 후 수집 도중 키가 레이트리밋/만료로 무효화 — 인덱싱 단계에서야 발견. 시스템은 인덱싱 시작 전 키 재검증을 수행하거나, 실패 시 bge-m3 fallback으로 전환하고 리포트에 "파이프라인 중간 백엔드 전환" 사실을 기록(검색3종 일관성 경고).
- **UC-1-X3**: 파이프라인 중간 단계 산출물이 이미 존재(재실행) — 단계별 산출물 캐시/재사용으로 D-4 시간 제약 내 부분 재실행이 가능해야 한다. 중복 임베딩 청크는 캐싱(NFR-4).

### Data Requirements
- **Input**: 임베딩 API 키(`.env`), 5개 도메인 소스 엔드포인트/URL, 도메인별 질의 세트, 청킹 정책(1000/100), RRF grid 후보.
- **Output**: S0 게이트 결정 기록, 도메인별 코퍼스 원본, 청크 메타데이터, BM25 인덱스, 임베딩 벡터 스토어, top-5 매트릭스, 정성 리포트(마크다운).
- **Side Effects**: 외부 API 소비(임베딩 호출, 코퍼스 다운로드), 로컬 파일·인덱스 산출물 생성. 서버 엔드포인트/DB 스키마 변경 없음. 코퍼스·키 파일은 커밋 제외.

## UC-2: 도메인별 코퍼스 3축 규모 스펙 산출

**Actor**: 강사(PoC 실행자) → 산출물 소비자: 강의 설계자(청킹·임베딩 비용·실습 시간 산정용)
**Preconditions**:
- 해당 도메인의 실코퍼스 수집이 완료되어 있다(UC-1 Primary 6 이후).
- 임베딩에 실제 사용할 모델의 토크나이저가 결정·로드되어 있다(S0 게이트 결과 기준).
- `kiwi` 형태소 분석기가 사용 가능하고 버전이 식별된다.

**Trigger**: 수집 완료 후 3축 스펙 산출 단계가 자동 실행되거나 강사가 spec 단계를 호출한다.

### Primary Flow (Happy Path)
1. 시스템이 도메인별 수집 문서 전체를 로드한다.
2. 각 문서의 **글자수**(한국어 음절 기준)를 계산하고 도메인 평균을 산출한다.
3. 각 문서를 S0에서 결정한 임베딩 모델 토크나이저로 토크나이즈해 **추정 토큰수**를 실측하고 도메인 평균을 산출한다(추정 공식이 아닌 토크나이저 실행 결과).
4. 각 문서를 `kiwi`로 형태소 분석해 **형태소 수**를 실측하고 도메인 평균을 산출한다.
5. 도메인별 **총 규모** = 평균값 × 전체 문서 개수를 글자/토큰/형태소 3축 각각 산출한다(FR-2.2).
6. 청킹 단계 산출 후 도메인별 **총 청크 개수**를 같은 표에 합산 보고한다(FR-3.3).
7. 표에 사용한 토크나이저 모델명·버전, `kiwi` 버전을 명시한다(FR-2.3/AC-2).
8. 5개 도메인 3축 스펙 표가 완성된다 — 평균 글자/평균 토큰/평균 형태소/전체 문서 개수/3축 총 규모/총 청크 개수.

**Postconditions**:
- AC-2를 충족하는 3축 스펙 표가 산출됨. 토크나이저·kiwi 버전이 표에 기록됨.
- 강의 설계자가 청킹·임베딩 비용·실습 시간을 산정할 수치 근거를 확보함.

### Alternative Flows
- **UC-2-A1: S0가 bge-m3 fallback으로 결정됨** — 토큰수 실측에 OpenAI/Claude 토크나이저 대신 `bge-m3` 토크나이저 사용.
  1. Primary 1~2 진행.
  2. 시스템이 `bge-m3` 토크나이저로 토큰수를 실측한다(백엔드 일관성 — UC-1-X1).
  3. 표에 "토크나이저 = bge-m3 (버전)"을 명시한다.
  4. Primary 4번 이후 동일.
- **UC-2-A2: 봇차단 도메인이 80건 미만** — 표본 규모가 작아도 3축 스펙은 산출.
  1. Primary 1~7 진행.
  2. 시스템이 해당 도메인 행에 "코퍼스 부족(N<80)" 플래그를 같이 표기한다.
  3. 평균값은 산출하되, 표본 규모 작음으로 인한 대표성 한계를 표 각주에 명시한다.

### Error Flows
- **UC-2-E1: 토크나이저 실행 실패** — API 토크나이저 호출 실패 또는 bge-m3 토크나이저 로드 실패.
  1. Primary 3 진행 중 토크나이저 실행 예외.
  2. 시스템이 3축 중 토큰수 축만 산출 실패로 표기하고, 글자/형태소 축은 계속 산출한다.
  3. 강사에게 토큰 축 누락 사유를 보고한다(부분 산출 — 비용 산정 영향 경고).
- **UC-2-E2: kiwi 형태소 분석 예외(빈 문서/제어문자)** — 정제 후에도 비정상 입력이 남은 경우.
  1. Primary 4 진행 중 특정 문서에서 kiwi 예외.
  2. 시스템이 해당 문서를 형태소 통계에서 제외하고 제외 건수를 표 각주에 기록한다.
  3. 제외 비율이 임계(예: 표본의 일정 비율) 초과 시 도메인 데이터 품질 경고를 보고한다.

### Edge Cases
- **UC-2-X1**: 문서 1개가 27,000자(D2 의약품 허가상세 전문약 전문) — 평균 글자수를 한쪽으로 끌어올리는 이상치. 표에 평균과 함께 최대/분포(또는 중앙값)를 병기해 이상치로 인한 왜곡을 드러내야 한다. 이 문서는 청킹 정책 경계 케이스로 UC-5와 연결된다.
- **UC-2-X2**: `kiwi`가 법령 전문용어/의약품명(예: "아토르바스타틴칼슘삼수화물")을 형태소로 못 쪼개고 미등록어로 1형태소 처리 — 형태소 수가 실제보다 과소 측정됨. 표 각주에 "도메인 특수 전문어 미등록어 처리로 형태소 수 과소 추정 가능"을 명시.
- **UC-2-X3**: 코퍼스 정제(HTML/캡션 제거) 후 본문이 사실상 비어 글자수 0에 수렴하는 문서 — 평균 계산에서 제외 또는 별도 카운트. 다수 발생 시 UC-7(pressReleaseView 함정) / UC-8(캡션 제거 경계)와 연결.
- **UC-2-X4**: D5 세출예산 CSV는 자연문이 아닌 표 데이터 — 글자/토큰/형태소 측정이 서술 문서와 의미가 다름. 표에 "구조화 표 기반 문서(서술형 아님)" 주석으로 비교 시 해석 주의를 명시.

### Data Requirements
- **Input**: 도메인별 정제 완료 코퍼스, 임베딩 토크나이저(모델명·버전), `kiwi`(버전), 청킹 결과(총 청크 개수).
- **Output**: 3축 스펙 표(평균 글자/토큰/형태소, 전체 문서 개수, 3축 총 규모, 총 청크 개수, 토크나이저·kiwi 버전, 이상치/제외 각주).
- **Side Effects**: 임베딩 토크나이저가 API형이면 토크나이즈 호출 비용 발생 가능(NFR-4 배치·캐싱 적용). 로컬형이면 비용 0.

## UC-3: 검색3종 실행 및 도메인 × 질의 top-5 매트릭스 산출

**Actor**: 강사(PoC 실행자) → 산출물 소비자: 강의 설계자(데모 질의 확정용)
**Preconditions**:
- 해당 도메인의 BM25(kiwi 토크나이저) 인덱스와 임베딩 인덱스가 구축되어 있다(UC-1 Primary 9 이후).
- 도메인별 질의 세트가 정의되어 있다 — 절반은 6차 검증 설계형(BM25/임베딩/하이브리드 우세 의도), 절반은 수강생이 던질 법한 거친 자연어형(FR-5.3).
- 하이브리드 RRF의 `k`·점수 정규화 grid 후보가 정의되어 있다.

**Trigger**: 인덱싱 완료 후 search 단계가 실행된다.

### Primary Flow (Happy Path)
1. 시스템이 도메인별 질의 세트를 로드한다(설계형 절반 + 자연어형 절반 구성 확인).
2. 각 질의에 대해 (a) BM25 검색을 실행해 top-5를 얻는다.
3. 각 질의에 대해 (b) 임베딩 검색을 실행해 top-5를 얻는다.
4. 각 질의에 대해 (c) 하이브리드를 RRF로 결합한다 — RRF `k`·정규화 grid를 순회하며 best 설정을 선택한다(FR-5.2).
5. 채택한 RRF `k`·정규화 설정값을 결과에 기록한다(AC-5).
6. 질의별로 세 방식의 top-5를 나란히 비교 가능한 매트릭스로 정렬한다(FR-5.4).
7. 5개 도메인 전체에 대해 도메인 × 질의 top-5 매트릭스가 완성된다.
8. 매트릭스 헤더/표지에 정성 한계 문구가 표기된다(AC-7).

**Postconditions**:
- AC-5를 충족하는 도메인 × 질의 top-5 매트릭스가 산출됨.
- 채택 RRF 설정 기록됨. 질의 세트 구성(설계형/자연어형 절반)이 확인 가능.
- UC-4 정성 판정의 입력 자료가 준비됨.

### Alternative Flows
- **UC-3-A1: RRF grid 중 best 선택** — 여러 `k`·정규화 조합이 후보일 때.
  1. Primary 4 진행.
  2. 시스템이 grid 각 조합으로 하이브리드 결과를 산출한다.
  3. 시스템(또는 강사 정성 판단)이 "설계형 질의에서 세 방식이 가장 명확히 갈리는" 조합을 best로 선택한다.
  4. best 조합 및 탈락 조합들의 요약을 결과에 기록한다(재현성).
- **UC-3-A2: bge-m3 임베딩 백엔드** — 임베딩 검색을 로컬 bge-m3 벡터로 수행.
  1. Primary 1~2 진행.
  2. 임베딩 검색을 bge-m3 인덱스로 실행한다(백엔드 일관성 — UC-1-X1).
  3. 매트릭스에 "임베딩 백엔드 = bge-m3"를 명시한다.
  4. Primary 4번 이후 동일.

### Error Flows
- **UC-3-E1: 임베딩 검색 단계 API 비용/레이트리밋 초과** — 질의 임베딩 호출이 한도 초과.
  1. Primary 3 진행 중 레이트리밋/쿼터 초과 응답.
  2. 시스템이 NFR-4 배치·캐싱·표본 상한을 재확인하고 재시도(백오프).
  3. 재시도 실패 시 임베딩 백엔드를 bge-m3로 전환하고 리포트에 "검색 단계 백엔드 전환" 일관성 경고를 기록한다(UC-1-X2 연계).
  4. 전환조차 불가하면 해당 도메인을 "임베딩/하이브리드 미실행, BM25만" 부분 산출로 표기.
- **UC-3-E2: 특정 질의가 어느 방식에서도 결과 0건** — 코퍼스에 매칭 문서 부재.
  1. Primary 2~4 진행 중 한 질의가 BM25·임베딩 모두 0건.
  2. 시스템이 해당 질의 행을 "결과 없음"으로 표기(빈 칸 아님, 명시적 0건).
  3. 질의가 코퍼스 범위를 벗어남(질의 부적절) 또는 코퍼스 부족 신호로 UC-4 판정에 전달.

### Edge Cases
- **UC-3-X1**: 코퍼스가 너무 작아(80건) 임베딩 top-5가 BM25 top-5와 사실상 동일 — 세 방식이 안 갈림. 매트릭스에 "세 방식 결과 중복도 높음(코퍼스 소규모 영향 의심)"을 표기하고 UC-4-X1로 전달한다.
- **UC-3-X2**: BM25 우세로 설계한 질의(`"중대재해처벌법 시행령"` 같은 정확 키워드형)인데 임베딩 top-5가 더 적절해 보임 — 의도와 반대 결과. 매트릭스에는 결과를 **있는 그대로** 기록하고, 해석/판정은 UC-4-X2에서 정성 기록(결과 왜곡 금지).
- **UC-3-X3**: 자연어형 질의가 수강생이 던질 법한 비문/오타("소상고인 알엔디 세금")를 포함 — BM25는 형태소 미스로 약하고 임베딩이 강할 수 있음. 매트릭스에 질의 원문 그대로 보존(질의 정제 금지 — 수강생 재현 환경 반영).
- **UC-3-X4**: D4(legalize-kr)는 짝 서술 문서 없이 법령·판례 단독 완결 — 검색3종 질의 세트가 다른 도메인과 구조가 다름(판례 전문 검색 중심). 매트릭스에 도메인 구조 차이 주석.
- **UC-3-X5**: 하이브리드 RRF grid의 모든 조합에서 BM25/임베딩 중 한쪽이 항상 지배 → 하이브리드가 의미를 못 만듦. best가 "하이브리드 무의미"라는 결론 자체를 기록(억지 best 선택 금지).

### Data Requirements
- **Input**: BM25 인덱스, 임베딩 인덱스, 도메인별 질의 세트(설계형+자연어형, 원문 보존), RRF `k`·정규화 grid.
- **Output**: 도메인 × 질의 top-5 매트릭스(세 방식 병렬), 채택 RRF 설정 기록, grid 탈락 조합 요약, 0건/중복 표기.
- **Side Effects**: 질의 임베딩 API 호출(NFR-4 배치·캐싱). bge-m3면 비용 0. 산출물 로컬 파일.

## UC-4: 도메인별 정성 Feasibility 판정 산출

**Actor**: 강사(PoC 실행자, 정성 판정 수행) → 산출물 소비자: 강의 설계자(도메인 교체·질의 보정 의사결정)
**Preconditions**:
- 도메인 × 질의 top-5 매트릭스가 산출되어 있다(UC-3 Primary 7 이후).
- 도메인별 코퍼스 건수·코퍼스 부족 플래그(있으면)가 확정되어 있다.
- 정답지(ground truth)는 **없다**(§1.3 — 본 PoC 전제).

**Trigger**: 검색3종 매트릭스 완성 후 report 단계가 실행된다.

### Primary Flow (Happy Path)
1. 강사가 도메인별로 매트릭스를 검토한다 — 설계형 질의에서 BM25/임베딩/하이브리드가 의도대로 갈리는지 정성 확인한다.
2. 강사가 자연어형 질의에서 수강생 관점의 결과 납득성을 정성 확인한다.
3. 강사가 도메인별로 `강의 데모 적합` / `조건부` / `부적합` 3단계 중 하나를 판정한다(FR-6.1).
4. 각 판정에 정성 근거를 첨부한다 — "어느 질의에서 세 방식이 갈렸는지/안 갈렸는지" 구체적으로 명시(FR-6.1, 리스크 표 "판정 주관성" 완화).
5. `조건부` 판정에는 강의 전 보정 액션(질의 교체, 코퍼스 보강, 필터 조정 등)을 구체적으로 명시한다(FR-6.2/AC-6).
6. `부적합` 판정에는 강의 데모에서 해당 도메인을 교체/제외해야 하는 사유를 명시한다.
7. 리포트 표지·각 도메인 섹션 헤더에 정성 한계 문구를 표기한다(FR-6.3/AC-7).
8. 5개 도메인 판정표가 완성된다 — 도메인 / 판정 / 정성 근거 / (조건부 시) 보정 액션 / (해당 시) 코퍼스 부족 플래그.

**Postconditions**:
- AC-6를 충족하는 도메인 판정표가 정성 근거와 함께 산출됨.
- 강의 설계자가 D-4~강의일 사이에 부적합 도메인 교체 또는 질의 보정을 결정할 근거를 확보함.
- AC-7 정성 한계 문구가 모든 산출물에 표기됨.

### Alternative Flows
- **UC-4-A1: 코퍼스 부족 도메인 판정** — 봇차단 소스 80건 미만(FR-1.2).
  1. Primary 1~2 진행(가능한 표본 범위 내에서).
  2. 강사가 판정을 내리되 "코퍼스 부족(N<80)으로 판정 신뢰도 저하" 단서를 명시한다.
  3. 보정 액션에 "코퍼스 보강 후 재판정 필요"를 우선 액션으로 명시한다(`조건부` 처리 권장).
- **UC-4-A2: 일부 도메인 수집/검색 실패로 판정 불가** — UC-1-E1/UC-3-E1 연계.
  1. 해당 도메인은 판정 3단계 대신 "판정 불가 — 사유(수집 실패/임베딩 미실행)"로 표기.
  2. 강의 설계자에게 해당 도메인 강의 사용 보류를 명시 권고.

### Error Flows
- **UC-4-E1: 정성 한계 문구 누락** — 리포트 산출물에 AC-7 문구가 빠짐.
  1. Primary 7 검증 단계에서 표지/헤더 문구 부재 감지.
  2. 시스템(또는 검증 체크)이 리포트 산출을 "AC-7 미충족"으로 차단/경고한다.
  3. 문구 삽입 후 재산출. (UC-9에서 별도 게이트로 검증)

### Edge Cases
- **UC-4-X1**: 세 방식이 안 갈림(임베딩 ≈ BM25, 코퍼스 80건 영향 의심 — UC-3-X1 연계) — 판정은 `조건부` 또는 `부적합`. 정성 근거에 "세 방식 미분화"를 명시하고, 보정 액션에 "코퍼스 보강 / 질의 재설계 / 도메인 교체" 중 선택지를 제시한다(강의 핵심 메시지 붕괴 위험 = 이 PoC가 잡으려는 바로 그 케이스, 리스크 표 참조).
- **UC-4-X2**: 설계 의도와 반대 결과(BM25 우세 설계 질의에서 임베딩이 더 적합 — UC-3-X2 연계) — 결과를 왜곡하지 않고 "설계 의도와 실결과 불일치"로 정성 기록. 보정 액션은 "질의를 실제 갈리는 형태로 교체" 또는 "강의 설명을 실결과에 맞게 수정". 의도대로 안 나왔다고 결과를 숨기지 않는다.
- **UC-4-X3**: 5개 도메인 중 다수가 `부적합`/`조건부` — 강의 데모 구성 자체 재검토 필요. 판정표 말미에 종합 권고(어느 도메인 조합으로 강의를 구성할지)를 추가한다.
- **UC-4-X4**: D4(legalize-kr) 단독 완결이라 검색3종 비교 구조가 타 도메인과 다름 — 판정 기준을 "법령·판례 검색 자체의 데모 성립성"으로 조정하고 그 기준 차이를 정성 근거에 명시.

### Data Requirements
- **Input**: 도메인 × 질의 top-5 매트릭스, 도메인별 코퍼스 건수·부족 플래그, 채택 RRF 설정, 질의 세트 구성 정보.
- **Output**: 도메인 판정표(판정 3단계 / 정성 근거 / 보정 액션 / 부족 플래그), 종합 권고, 정성 한계 문구.
- **Side Effects**: 없음(판정은 기존 산출물 기반 정성 분석). 리포트 마크다운 파일만 생성.

## UC-5: 청킹 정책 적용 및 통짜 임베딩 방지

**Actor**: 강사(PoC 실행자, 파이프라인 chunk 단계)
**Preconditions**:
- 도메인별 정제 코퍼스 수집 완료(UC-1 Primary 6 이후).
- 청킹 정책이 1000토큰 슬라이딩 윈도우 / overlap 100토큰으로 강제 설정되어 있다(FR-3.1).
- 토큰 계산용 토크나이저가 S0 결정 백엔드와 일관되게 로드되어 있다.

**Trigger**: 3축 스펙 산출 후(또는 인덱싱 직전) chunk 단계가 실행된다.

### Primary Flow (Happy Path)
1. 시스템이 도메인별 문서를 순회한다.
2. 각 문서를 1000토큰 슬라이딩 윈도우·overlap 100토큰으로 청크 분할한다(FR-3.1).
3. 단일 문서를 통짜 1개 임베딩 벡터로 만들지 않음을 보장한다 — 1000토큰 초과 문서는 반드시 ≥2 청크로 분할(FR-3.2).
4. 각 청크에 문서ID·청크ID·문자 오프셋 메타데이터를 부여한다.
5. 도메인별 총 청크 개수를 집계해 3축 스펙 표(UC-2)로 전달한다(FR-3.3).
6. 청킹 결과가 인덱싱(UC-1 Primary 9)으로 전달된다.

**Postconditions**:
- AC-3 충족 — 1000/100 청킹 적용, 통짜 단일 임베딩 없음을 산출물에서 검증 가능.
- 청크 메타데이터(문서ID·청크ID·오프셋) 생성됨. 도메인별 총 청크 개수 기록됨.

### Alternative Flows
- **UC-5-A1: 1000토큰 이하 짧은 문서** — 분할 불필요.
  1. Primary 1~2 진행.
  2. 문서 토큰수 ≤ 1000이면 단일 청크로 처리하되, 이는 "통짜 임베딩 금지(FR-3.2)" 위반이 아님(문서 자체가 청크 한도 이내).
  3. 메타데이터에 "단일 청크(문서≤1000토큰)"로 표기해 X1 위반과 구분.

### Error Flows
- **UC-5-E1: 토크나이저 미가용으로 토큰 경계 산출 불가** — 청크 경계를 토큰 기준으로 못 자름.
  1. Primary 2 진행 중 토크나이저 예외.
  2. 시스템이 청킹을 중단하고(글자수 근사 폴백은 토큰 정책 위반이므로 금지) 강사에게 보고.
  3. 토크나이저 복구 후 재실행. (청킹은 멱등 재실행 가능해야 함)

### Edge Cases
- **UC-5-X1**: D2 의약품 허가상세 전문약 27,000자+ 단일 문서(UC-2-X1 연계) — 토큰 한도(1000) 대비 수십 배. 슬라이딩 윈도우로 다수 청크 생성되어야 하며, 통짜 1벡터 금지(FR-3.2)가 바로 이 케이스에서 검증된다. 청크 수가 비정상적으로 많을 때 메타데이터로 추적 가능해야 한다.
- **UC-5-X2**: 청킹 후에도 단일 청크가 여전히 토큰 한도 초과 — overlap 경계 처리 버그 또는 분리 불가 토큰열(긴 표/코드성 텍스트). 시스템은 한도 초과 청크를 강제 재분할하거나 "한도 초과 청크" 플래그로 격리하고 리포트에 보고(통짜 임베딩 금지 원칙 유지).
- **UC-5-X3**: overlap 100토큰이 짧은 문서(예: 150토큰)에 적용될 때 중복 과다 — 청크 간 거의 동일. 중복 청크 캐싱(NFR-4)으로 임베딩 비용 중복 제거, 메타데이터로 중복성 추적.
- **UC-5-X4**: D5 세출예산 CSV 행 기반 데이터 — 토큰 슬라이딩 윈도우가 표 구조를 임의 절단. 청킹 단위 정책(행 단위 vs 토큰 윈도우) 차이를 메타데이터/리포트에 주석(서술 문서와 동일 정책 적용 시 한계 명시).

### Data Requirements
- **Input**: 도메인별 정제 코퍼스, 토크나이저(S0 백엔드 일관), 청킹 정책(1000/100 고정).
- **Output**: 청크 + 메타데이터(문서ID·청크ID·오프셋), 도메인별 총 청크 개수, 한도 초과/중복 플래그.
- **Side Effects**: 로컬 청크 산출물 파일. 통짜 단일 벡터 금지 불변식이 인덱싱 단계 입력에 반영됨.

## UC-6: 코퍼스 수집 — 봇차단·HTTP 실패·소스 폐기 처리

**Actor**: 강사(PoC 실행자, 파이프라인 collect 단계)
**Preconditions**:
- S0 게이트 통과(UC-1 Primary 1~3).
- 5개 도메인 소스 엔드포인트/URL이 정의되어 있다.
- 봇차단 소스(D2 의약품 허가상세, D1/D3 정책뉴스)에 대한 "최소 80건" 가변 목표와 일반 소스의 300~500건 목표가 설정되어 있다.

**Trigger**: collect 단계 실행(스파이크 후 본 수집).

### Primary Flow (Happy Path)
1. 시스템이 도메인별로 스파이크(소량) 수집을 먼저 시도해 경로 유효성을 확인한다.
2. 일반 소스(D5 무인증 data.go.kr, D1 상가정보, D2 약가마스터, D3 국민연금사업장, D4 legalize-kr git clone)는 300~500건 목표로 수집한다.
3. 봇차단 소스는 최소 80건 가변 목표로 수집한다(FR-1.2).
4. 정책뉴스는 `policyNewsView` 경로를 사용하고 `pressReleaseView`는 사용하지 않는다(FR-1.3 — UC-7 함정).
5. 본문 사진 캡션·HTML 태그를 정규식으로 제거한다(FR-1.4 — UC-8 경계).
6. 정책뉴스 소상공인/고용·금융 서브셋은 `article_body` 글자수 ≥ 800 필터를 적용한다(FR-1.1).
7. D4 legalize-kr는 수집 시점 commit 해시를 고정·기록한다(NFR-6 — UC-6-X3).
8. 도메인별 수집 건수를 기록한다(AC-1).

**Postconditions**:
- 도메인별 정제 코퍼스와 수집 건수 기록 확보. 봇차단 소스는 ≥80건 또는 "코퍼스 부족" 플래그.
- legalize-kr commit 해시 스냅샷 고정.

### Alternative Flows
- **UC-6-A1: 봇차단 소스 300건 미달이나 80건 이상** — 가변 목표 충족.
  1. Primary 3 진행, 봇차단으로 300건 미달.
  2. 수집 건수가 80건 이상이면 "가변 목표 충족(N건)"으로 정상 진행(FR-1.2).
  3. 해당 도메인 코퍼스 부족 플래그 없이 진행하되, 표본 규모를 리포트에 명시.
- **UC-6-A2: legalize-kr 2.7G 대용량** — 전체 clone 부담.
  1. Primary 7 진행 전, legalize-kr 판례 데이터가 2.7G로 D-4 시간 제약 위협.
  2. 시스템이 sparse-checkout 또는 사건종류 1개 샘플로 부분 수집한다(NFR-1 표본 상한).
  3. 부분 수집 범위(어느 사건종류/경로)와 commit 해시를 함께 기록한다(재현성).

### Error Flows
- **UC-6-E1: HTTP 차단/타임아웃** — 봇차단 또는 일시적 네트워크 실패.
  1. Primary 1~3 진행 중 403/429/타임아웃 응답.
  2. 시스템이 백오프 재시도(레이트 조절, User-Agent/요청 간격 조정 범위 내).
  3. 재시도 후에도 80건 미달이면 UC-6-E3로 전이.
- **UC-6-E2: 소스 데이터 폐기/스키마 변경** — 조달청 사례처럼 데이터가 제공 중단됨.
  1. Primary 1 스파이크에서 응답은 오나 본문/필드가 비거나 폐기 안내 페이지.
  2. 시스템이 "소스 폐기/스키마 변경"으로 분류하고 재시도하지 않는다(재시도 무의미).
  3. 해당 도메인을 "수집 실패 — 소스 폐기"로 기록하고 UC-1-E1(도메인 제외 HITL)로 전이.
- **UC-6-E3: 봇차단 소스 80건 최소 목표도 미달** — 가변 목표조차 실패.
  1. UC-6-E1 재시도 후에도 80건 미만.
  2. 시스템이 해당 도메인에 "코퍼스 부족(N<80)" 플래그를 강제 부여한다(FR-1.2/AC-1).
  3. 파이프라인은 중단하지 않고 진행 — 해당 도메인은 UC-4-A1(코퍼스 부족 판정)로 전달.
- **UC-6-E4: D5 무인증 data.go.kr CSV 다운로드 실패** — 무인증 소스조차 다운로드 불가.
  1. Primary 2 진행 중 CSV 다운로드 실패(파일 이동/포맷 변경).
  2. 시스템이 정책브리핑 보도자료 본문 단독으로 D5를 구성할지 강사에게 보고(HITL).
  3. 결정에 따라 D5 코퍼스 구성 변경을 리포트에 기록.

### Edge Cases
- **UC-6-X1**: 정책뉴스 `article_body` 글자수 ≥ 800 필터 후 잔존 건수 < 80 — 필터가 코퍼스를 과도 축소. 필터 임계와 잔존 건수를 리포트에 명시하고 "코퍼스 부족" 판단에 반영(필터 완화 vs 부족 플래그 트레이드오프 기록).
- **UC-6-X2**: 정책뉴스 본문에 사진 캡션 `(ⓒ뉴스1...)`이 다양한 변형(`(c)`, `ⓒ`, 전각/반각)으로 혼입 — UC-8 정규식 경계로 위임. 수집 단계에서는 원문 보존, 정제는 별도 단계에서 검증.
- **UC-6-X3**: legalize-kr가 수집 후 강의 전 force-push로 commit 해시 변동 — 스냅샷 고정값과 강의 시점 불일치. 수집 시점 commit 해시를 리포트에 명시하고, 강의 재현 시 동일 해시 checkout을 전제로 안내(NFR-6, 리스크 표).
- **UC-6-X4**: D2 약가마스터 `품목기준코드`와 의약품 허가상세 JOIN 키 불일치(코드 체계 변경) — JOIN 매칭률 저하. 매칭률을 리포트에 기록하고, 매칭 실패 문서의 검색 코퍼스 포함 여부 정책을 명시.
- **UC-6-X5**: 동일 기사/문서 중복 수집(페이지네이션 중복, 동일 보도자료 재게시) — 중복 제거 전후 건수를 함께 기록. 중복 제거 후 80건 미달 시 UC-6-E3 적용.

### Data Requirements
- **Input**: 도메인 소스 엔드포인트/URL, 수집 목표(300~500 / 최소 80), 필터(article_body≥800), 정규식(캡션/HTML), legalize-kr 대상 경로.
- **Output**: 도메인별 정제 코퍼스 원본, 수집 건수, 코퍼스 부족 플래그, legalize-kr commit 해시, 중복 제거 통계.
- **Side Effects**: 외부 소스 다운로드(data.go.kr/korea.kr/의약품안전나라/GitHub). 코퍼스 파일은 `.gitignore` 제외(NFR-5/AC-8).

## UC-7: 정책뉴스 본문 추출 경로 함정(pressReleaseView) 방지

**Actor**: 강사(PoC 실행자, collect 단계 내 정책뉴스 추출 로직)
**Preconditions**:
- 정책뉴스 소스(D1 소상공인 서브셋, D3 고용·금융 서브셋) 수집 대상이다.
- 6차 검증에서 `pressReleaseView`는 본문이 HWP-iframe에 갇혀 ~40자만 추출됨이 실측 확인되었다(함정 #2).

**Trigger**: 정책뉴스 본문 추출 로직 실행.

### Primary Flow (Happy Path)
1. 시스템이 정책뉴스 본문을 `policyNewsView` 경로로 요청한다(FR-1.3).
2. `article_body` 필드에서 본문을 추출한다.
3. 추출 본문 글자수가 합리적 범위(≥800 필터 대상)인지 확인한다.
4. 캡션·HTML 제거(UC-8) 후 코퍼스에 포함한다.

**Postconditions**:
- 정책뉴스 본문이 전문 글자수로 정상 수집됨(40자 절단 함정 회피).

### Error Flows
- **UC-7-E1: 코드가 `pressReleaseView`를 잘못 사용** — 함정 #2 발생.
  1. 추출 로직이 `pressReleaseView` 엔드포인트를 호출한다.
  2. 본문이 HWP-iframe에 갇혀 ~40자만 추출된다.
  3. 시스템이 본문 글자수가 비정상적으로 짧음(예: <100자 다수)을 감지하고 "추출 경로 의심 — pressReleaseView 함정" 경고를 발생시킨다.
  4. 강사가 `policyNewsView` 경로로 전환한다(Primary로 복귀).
- **UC-7-E2: policyNewsView도 본문 단축** — 경로는 맞으나 일부 기사 본문이 짧음.
  1. Primary 1~3 진행, 일부 기사 `article_body` < 800.
  2. ≥800 필터(FR-1.1)로 자연 제외. 함정과 구분하기 위해 "정상 필터 제외 vs 추출 경로 함정"을 구분 로깅.

### Edge Cases
- **UC-7-X1**: `policyNewsView` 사용했으나 평균 본문 글자수가 40자 부근에 군집 — 함정 #2의 변형(엔드포인트는 맞으나 파싱 셀렉터 오류). 추출 본문 글자수 분포를 리포트에 기록해 40자 절단 패턴을 조기 감지(이 PoC가 인덱싱·검색 전에 잡아야 할 함정).
- **UC-7-X2**: 본문 추출은 정상이나 ≥800 필터로 코퍼스가 80건 미달 — UC-6-X1과 동일 트레이드오프로 위임.

### Data Requirements
- **Input**: 정책뉴스 기사 식별자, `policyNewsView` 엔드포인트.
- **Output**: 정상 추출 본문(전문), 본문 글자수 분포 로그, 함정 감지 경고(발생 시).
- **Side Effects**: korea.kr 요청. 잘못된 경로 사용 시 코퍼스 품질 붕괴(이 UC가 방지 대상).

## UC-8: 본문 노이즈 정제 — 사진 캡션·제3자 저작권 표기 제거 경계

**Actor**: 강사(PoC 실행자, collect 단계 내 정제 로직)
**Preconditions**:
- 정책뉴스/정책브리핑 본문이 수집되어 있다(UC-7 Primary 이후).
- 사진 캡션 `(ⓒ뉴스1...)` 등 제3자 저작권 표기 및 HTML 태그 제거 정규식이 정의되어 있다(FR-1.4).

**Trigger**: 정제 로직 실행.

### Primary Flow (Happy Path)
1. 시스템이 본문에서 HTML 태그를 정규식으로 제거한다.
2. 사진 캡션·제3자 저작권 표기 패턴(`(ⓒ뉴스1...)` 등)을 정규식으로 제거한다(라이선스·노이즈 제거).
3. 정제 후 본문을 코퍼스에 포함한다.
4. 정제 전후 글자수 변화를 로깅한다(과다 제거 감지용).

**Postconditions**:
- 본문이 라이선스 안전·노이즈 제거 상태로 코퍼스에 포함됨.

### Error Flows
- **UC-8-E1: 캡션 정규식 미스 — 캡션이 본문에 잔존** — 패턴 변형 미커버.
  1. Primary 2 진행, `(c)뉴스1`/`ⓒ 연합뉴스`/전각·반각 변형이 정규식에 안 걸림.
  2. 정제 후 본문에 제3자 저작권 표기 잔존(라이선스 리스크 + 검색 노이즈).
  3. 정제 전후 글자수 변화가 미미함을 감지해 "캡션 제거 누락 의심" 경고.
- **UC-8-E2: 정규식 과다 제거 — 본문 손상** — 캡션 패턴이 본문 일부와 충돌.
  1. Primary 2 진행, 정규식이 본문 괄호 구절(`(예: ...)` 등)까지 제거.
  2. 정제 후 글자수가 비정상적으로 급감(예: 30% 이상 감소)을 감지해 "과다 제거 의심" 경고.
  3. 강사가 정규식 경계를 조정 후 재정제.

### Edge Cases
- **UC-8-X1**: 사진 캡션 `(ⓒ뉴스1...)`이 본문 중간(문장 사이)에 삽입 — 제거 후 문장 연결이 부자연(검색 청크 품질 영향). 제거 후 공백 정규화로 문장 연결을 보정하고, 보정 규칙을 정제 로직에 명시.
- **UC-8-X2**: 캡션 표기 변형 다양성(`ⓒ`, `(c)`, `©`, `[사진=...]`, `▲...제공`) — 단일 정규식으로 전부 커버 불가. 커버한 패턴 목록과 미커버 가능성을 리포트에 명시(완전성 한계 정직 기록).
- **UC-8-X3**: 정제로 본문이 사실상 비어 글자수 0 수렴(UC-2-X3 연계) — 캡션·태그 외에 본문 실체가 없던 문서. ≥800 필터/평균 계산에서 제외 처리하고 제외 사유를 "정제 후 본문 부재"로 기록.

### Data Requirements
- **Input**: 원문 본문, 캡션/HTML 제거 정규식 패턴 목록.
- **Output**: 정제 본문, 정제 전후 글자수 변화 로그, 과다/과소 제거 경고, 커버 패턴 목록.
- **Side Effects**: 없음(텍스트 정제). 정제 품질이 UC-2 3축 스펙·UC-3 검색 품질에 직접 전파.

## UC-9: 정성 한계 문구 표기 게이트 (AC-7 강제)

**Actor**: 강사(PoC 실행자) / 시스템 검증 게이트
**Preconditions**:
- 정성 리포트 산출물(3축 스펙 표, top-5 매트릭스, 도메인 판정표)이 생성 단계에 있다.
- §1.3 정성 한계 문구 = "본 결과는 정성 feasibility 판정이며 정량 검색 품질 평가가 아님".

**Trigger**: 모든 리포트 산출물 생성 시점(report 단계 마무리).

### Primary Flow (Happy Path)
1. 시스템이 각 리포트 산출물의 표지/헤더를 검사한다.
2. 표지·각 도메인 섹션 헤더에 정성 한계 문구가 존재함을 확인한다(FR-6.3/AC-7).
3. 모든 산출물이 문구를 포함하면 게이트 통과.
4. 리포트가 최종 산출물로 확정된다(`poc/` 하위 커밋 대상).

**Postconditions**:
- AC-7 충족 — 모든 산출물에 정성 한계 문구 표기 확인.

### Error Flows
- **UC-9-E1: 일부 산출물 문구 누락** — 매트릭스 또는 판정표 헤더에 문구 부재.
  1. Primary 2에서 누락 산출물 감지.
  2. 게이트가 "AC-7 미충족 — 산출물 N 문구 누락"으로 리포트 확정을 차단한다.
  3. 문구 삽입 후 게이트 재실행(멱등).

### Edge Cases
- **UC-9-X1**: 문구가 존재하나 표현이 변형되어 "정량 평가가 아님" 의미가 약화 — 게이트는 핵심 의미 키워드(정성 feasibility / 정량 평가 아님)를 모두 포함하는지 검사. 의미 약화 변형도 미충족 처리.
- **UC-9-X2**: 산출물이 여러 파일로 분할(스펙/매트릭스/판정표 별도 파일) — 각 파일 표지에 독립적으로 문구가 있어야 함(한 파일에만 있으면 미충족).

### Data Requirements
- **Input**: 리포트 산출물 전체(파일 목록), 정성 한계 문구 표준 텍스트.
- **Output**: 게이트 통과/차단 결과, 누락 산출물 목록(차단 시).
- **Side Effects**: 미충족 시 리포트 확정·커밋 차단. 통과 시 `poc/` 커밋 가능 상태로 전이.

## UC-10: 배포 안전 — 코퍼스·키 커밋 제외 검증 (AC-8 강제)

**Actor**: 강사(PoC 실행자) / 커밋 전 검증
**Preconditions**:
- PoC가 공개 배포 리포 `aiceo-4th-training`의 `poc/` 하위에 위치한다(NFR-5).
- 수집 코퍼스 데이터, 인덱스 산출물, API 키(`.env`)가 로컬에 생성되어 있다.
- `.gitignore`에 코퍼스 데이터·인덱스·`.env` 패턴이 등록되어 있다.

**Trigger**: PoC 결과 커밋 직전.

### Primary Flow (Happy Path)
1. 강사가 커밋 대상 파일 목록을 확인한다.
2. 코퍼스 원본, 청크 메타데이터, BM25 인덱스, 임베딩 벡터 스토어, `.env`가 커밋 대상에서 제외됨을 확인한다(`.gitignore` 적용).
3. PoC 코드와 정성 리포트(마크다운)만 커밋 대상임을 확인한다(AC-8).
4. `poc/` 하위로 커밋한다.

**Postconditions**:
- AC-8 충족 — 코퍼스·키 미커밋, 코드·정성 리포트만 커밋.

### Error Flows
- **UC-10-E1: 코퍼스/키가 커밋 스테이징에 포함** — `.gitignore` 누락 또는 강제 add.
  1. Primary 2에서 코퍼스 파일/`.env`가 스테이징에 포함됨을 감지.
  2. 검증이 커밋을 차단하고 해당 파일을 명시 보고.
  3. `.gitignore` 보강 또는 unstage 후 재검증. (공개 배포 리포이므로 키/데이터 유출 = 치명 — 차단 필수)

### Edge Cases
- **UC-10-X1**: 정성 리포트 본문에 수집 원문 일부(저작권 본문·캡션 잔존)가 인용 포함 — 리포트는 커밋되므로 라이선스 리스크. 리포트 내 인용은 검색 결과 식별 가능한 최소 스니펫으로 제한하고, 제3자 저작권 본문 전문 인용 금지(UC-8 정제와 연계).
- **UC-10-X2**: 인덱스 산출물이 코퍼스 원문을 역추출 가능한 형태로 포함(예: 청크 텍스트 저장 벡터 스토어) — 인덱스도 코퍼스로 간주해 커밋 제외 대상에 포함(`.gitignore` 패턴이 인덱스 디렉터리를 포함하는지 검증).

### Data Requirements
- **Input**: 커밋 스테이징 파일 목록, `.gitignore` 패턴.
- **Output**: 검증 통과/차단 결과, 위반 파일 목록(차단 시).
- **Side Effects**: 위반 시 커밋 차단. 통과 시 코드·정성 리포트만 공개 리포에 반영.

## 부록 B: 정성 한계 불변식 (모든 UC 공통)

본 문서의 모든 UC는 다음 불변식을 위반하지 않는다(§1.3 스코프 정직성):

1. 어떤 UC도 nDCG/MRR/Recall@k 등 정량 검색 품질 지표를 산출 단계로 포함하지 않는다(§1.7 비목표).
