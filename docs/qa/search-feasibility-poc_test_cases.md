> Based on [PRD](../PRD.md) §1 (AC-1~AC-8) and [Use Cases](../use-cases/search-feasibility-poc_use_cases.md) (UC-1~UC-10 + 부록 A 추적매트릭스 + 부록 B 정성 불변식)

## 0. 이 문서의 사용 규칙 (필독 — PoC 특유 QA 전략)

본 PoC는 **웹 서비스가 아니라 CLI 검증 파이프라인 + 외부 의존**이다. 일반 웹 QA와 다르게 다음 전략을 적용한다.

### 0.1 자동화 가능 여부 3분류

| 분류 | 의미 | 검증 방법 |
|------|------|-----------|
| **자동** | 코드/CI로 pass·fail 판정 가능 (산출물 존재·형식·수치·gitignore 등) | 단위/통합 테스트, assert, `git check-ignore` 등 |
| **반자동** | 자동으로 산출물·로그를 만들고, 형식·임계는 자동 검증하되 최종 결론은 사람 판단 보조 | fixture로 산출물 생성 → 자동 형식 검증 + 사람 체크리스트 |
| **수동-육안** | 정답지 없는 정성 결론(검색3종이 의도대로 갈렸나, 판정 타당성) — 자동 판정 불가 | 강사 육안 리뷰 + 명시된 체크리스트 |

### 0.2 외부 의존 격리 (실네트워크 vs fixture)

5개 수집 소스는 실네트워크 의존이다: ① data.go.kr 무인증 fileData(상가/약가/세출예산), ② korea.kr `policyNewsView`/정책브리핑, ③ 의약품안전나라 허가상세, ④ GitHub `legalize-kr` git clone, ⑤ 임베딩 API(OpenAI/Claude).

- **실수집 요구 TC** (`[NET]` 표기): 실제 네트워크·실소스 응답이 있어야만 검증 가능. CI 비결정적 — 강사가 수동/스케줄 실행. 라이브 스모크 한정.
- **fixture/mock 검증 가능 TC** (`[FX]` 표기): 저장된 응답 스냅샷·합성 코퍼스로 로직(파싱/정제/청킹/스펙/RRF/게이트)을 결정적으로 검증. CI 대상.
- 원칙: **수집 경로 유효성(스파이크)·소스 가용성은 `[NET]`, 그 외 모든 가공·검증 로직은 `[FX]`로 분리**한다. `[NET]` 실패가 `[FX]` 로직 회귀를 가리지 않게 한다.

### 0.3 known-answer fixture (정성 산출물의 자동 검증 한계 보완)

- 3축 스펙(글자/토큰/형태소 × 문서개수), 청킹 경계, RRF 결합, 정규식 정제는 **작은 합성 fixture로 known-answer 테스트**가 가능하다 → `[FX]` 자동.
- "검색3종이 의도대로 갈렸나", "feasibility 판정이 타당한가"는 정답지가 없어 **자동 판정 불가** → 수동-육안. 단, 산출물의 **존재·형식·구성**(예: top-5 매트릭스가 세 방식 병렬 형태인가, 정성 한계 문구가 있는가)은 자동 검증한다.

### 0.4 우선순위

- **P0**: 실패 시 강의 데모가 무너지거나 배포 안전 사고(키/저작권 유출). 머지 차단.
- **P1**: 핵심 산출물 정확성·재현성. 머지 전 통과 권장.
- **P2**: 엣지·견고성. 추적은 하되 부분 미충족 시 리포트 명시로 갈음 가능.

### 12.4 갭 및 한계 명시

1. **자동 판정 불가 영역(설계상 한계, 갭 아님)**: "검색3종이 의도대로 갈렸나", "feasibility 판정이 타당한가"는 정답지(ground truth) 부재로 자동 판정 불가(§1.3, 부록 B). TC-4.5를 **수동-육안 체크리스트**로 명문화해 갈음. 자동 영역은 산출물 존재·형식·구성·플래그로 한정. 이는 PoC 스코프 정직성에 따른 의도된 한계이며 미커버 갭이 아님.
2. **`[NET]` TC의 CI 비결정성**: TC-1.8, TC-6.1, TC-6.4는 실네트워크·실소스 응답 의존으로 CI 비결정적. 강사 수동/스케줄 라이브 스모크로 운영하고, 해당 로직의 결정적 검증은 `[FX]` 스냅샷 TC(TC-7.1, 8.x 등)가 담당해 회귀 안전망 분리.
3. **NFR 직접 인수 기준 부재**: NFR-1(D-4 시간)·NFR-3(한국어 처리)은 독립 AC가 없어 관련 FR/AC TC에 내재 검증(NFR-1→TC-1.11 부분 재실행·TC-6.4 표본 상한 / NFR-3→TC-2.1·5.1·7.1 kiwi·토큰). 단독 정량 SLA TC는 비목표(§1.7)로 제외.
4. **legalize-kr force-push 시점 의존(UC-6-X3/TC-6.11)**: 강의 시점 실제 force-push 발생 여부는 PoC 통제 밖. TC는 "수집 시점 해시 기록 + 재현 안내 문구 존재"까지만 자동 검증 가능(NFR-6 범위).
5. **D5 세출예산 CSV 비서술 데이터(TC-2.11/5.9/UC-3 D5)**: 표 데이터에 서술 문서 전제 정책(글자/토큰/형태소·토큰 윈도우 청킹) 적용 시 해석 한계는 주석으로 노출만 가능, 정량 동등 비교는 불가(설계상 한계 명시).

## 13. 테스트 작성자(test-writer)에게

- `[FX]` TC를 우선 구현(결정적·CI 대상). 합성 fixture는 **known-answer**(사전 계산 글자/토큰/형태소·청크 경계)로 작성해 회귀 잠금.
- `[NET]` TC는 라이브 스모크로 분리하고 CI 게이트에서 제외(비결정성). 실패 시 강사 보고.
- TC-7.3(pressReleaseView 정적 금지), TC-10.5(`.gitignore` 패턴 존재)는 정적 검사 — 가장 싸고 강한 회귀 잠금이므로 최우선.
- 수동-육안 TC(TC-4.5)는 자동 테스트로 만들지 말고 **체크리스트 문서/리포트 템플릿**으로 산출물에 내장(강사 서명란 포함).
- 부록 B 불변식(§11)은 모든 산출물에 걸친 cross-cutting 회귀 — 전 파이프라인 통합 테스트의 사후 단언으로 배치.
